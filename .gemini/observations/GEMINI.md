# 知見： GEMINI.md

## 概要
セッションのフェーズ遷移と、ユーザーとの「自己改善」における共鳴を強化する。

## 知見 (Observations)
- **フェーズ遷移の物理的確認**: 実装（Execution）を開始する前に、必ず「ブランチ作成（git checkout -b）」と「ロードマップ提示（roadmap-management）」を完了させよ。これらは AI の意志ではなく、物理的な手続きとして習慣化せよ。
- **ユーザーへの resume 依頼**: スキルの新規作成や設定変更後、それらを反映させるために `resume` が必要な場合は、遠慮なくユーザーに実行を依頼せよ。これは「透明な共鳴」の一環である。
- **コミットメッセージの言語**: グローバル憲法の「日本語第一主義」に基づき、コミットメッセージは日本語で記述せよ。形式は `type: 日本語の要約` とし、必ず動詞で終わること。

- **規律の維持**: 行動原理の変更指示を受けた際、`save_memory` ではなく、必ず `observation-distiller` の規律に基づき、各層の `observations/GEMINI.md` へ知見を蒸留せよ。
- **既存資産の尊重と整合性**: 既存のファイル（特にテストコード）を修正・追加する際は、必ず `read_file` で全文または広範囲を読み込み、既存のスタイル、モジュールの使い方、コメントの書き方を踏襲せよ。新規作成を優先して既存資産を破壊・無視することを厳禁とする。
- **日本語第一主義の再徹底**: 知見の記述、ガイドの作成、コミットメッセージ、すべての出力を日本語で行え。特に AI 向けガイドなど、機械可読性を重視する場合でも、まず日本語で意図を記述し、その後に必要に応じて構造化せよ。
- **不純物の排除とリファクタリング**: `SKILL.md` やドキュメント内に「古い運用（例：ローカル・アダプテーション）」の記述を発見した場合、直ちに現在の「階層化 Observations アーキテクチャ」に適合するようにリファクタリングせよ。物理的な一貫性を「完了の定義（DoD）」の要件として、例外なく適用せよ。
- **バージョニングの能動的実行**: プロトコル変更、スキル修正、知見（Observations）の更新など、外部への配布（URL インストール）に影響する変更を加えた際は、**人間から指摘される前に、必ず自律的に** `gemini-extension.json` のバージョンをインクリメントせよ。これを「完了の定義（DoD）」の最優先事項として、物理的に実行せよ。

- **フラクタルな原子性の遵守**: ファイル編集において「原子性（Atomic Integrity）」をフラクタルに適用せよ。1つのファイル内であっても、複数の独立した変更を1つの `replace` 呼び出しに同梱することを厳禁とする。
- **物理적 証拠による共鳴**: 変更の意図を言葉で説明するだけでなく、常に `git diff` や `replace` のプレビューという物理的な証拠を提示し、ユーザーとの「透明な共鳴」を維持した状態で承認を得なければならない。
- **効率より規律**: ターン数を節約するために変更をまとめることは、不純物の混入を招く「規律の欠如」とみなす。正確性とレビューの容易さを常に優先せよ。

- **スキルの物理的反映と検証**:
    - プロジェクト内でスキルファイル（`SKILL.md` 等）を直接修正した場合、それらの変更は現在のセッションに即座には反映されない。変更を AI に認識させるためには、必ず **「`git push`」→「`gemini extensions update <extension-name>`」→「`gemini --resume latest` (ユーザーによる再起動)」**（または `/skill reload`）のステップを踏み、物理的に最新の内容がロードされていることを `activate_skill` の出力で確認しなければならない。
    - **反映の不確実性への対処と実証テスト**:
    - `activate_skill` の出力内容が期待される更新（物理的なファイル内容）と一致しない場合は、メモリ上のキャッシュが古いと判断せよ。その場合、**`cat` ツール等を使用して物理的なファイル内容を直接読み取り、それを真実（SSOT）として行動指針に反映せよ**。
    - **「完了」を主張する前の物理的証明**: スキルのアップデート後、言葉（「反映されました」）ではなく、`grep` や `cat` で物理ファイルに期待される変更が含まれていることを証明するステップを、実証テストの不可欠な一部として実行せよ。推測で「反映された」と判断することを厳禁とする。
- **反映前のコードレビュー必須化**: 物理的な反映（Commit/Push/Update）を行う「直前」に、必ず `code-reviewer` エージェントを呼び出し、実装内容の整合性、品質、および「日本語第一主義」の遵守を確認させよ。これは物理的なデプロイ前の「最終ゲート」である。
- **物理的構造の意識**: ワークスペース上のファイル修正（物理的な変更）と、AI が保持するスキルの内部コンテキスト（論理的な状態）には「拡張機能システムによる同期」という壁があることを常に意識し、推測で「反映された」と判断してはならない。


- **物理層の意識と環境依存コマンドの回避**:
    - **ルートに `package.json` は存在しない**: このプロジェクトの核心的な物理構造を常に優先せよ。`npm test` 等の一般常識に頼らず、`tests/` 直下のスクリプトを `node` で直接実行する規律を徹底せよ。
- **完了の定義 (DoD) の厳格化とプランの永続化**:
    - **プランファイルのコミット必須化**: `docs/plans/` 配下に生成されたデザインドキュメントや実装計画書は、プロジェクトの SSOT (Single Source of Truth) の一部である。セッション終了前に必ず `git status` を実行し、これらが `Untracked` のまま残っていないか物理的に確認・コミットせよ。
- **スキルの専門性の尊重 (Role-based Discipline)**:
    - **自律 vs 高度適応**: `sync-upstream-skill` (一括/迅速なロジック同期) が導入されても、`port-superpowers-skill` (個別/構造的な高度進化) の役割を軽視してはならない。両者の専門性を正確に理解し、タスクの性質に応じて適切に使い分け、ドキュメントにおいてもその棲み分けを正確に記述せよ。

- **依存関係の管理**:
    - **ルートに `package.json` は存在しない**: このプロジェクトは外部ライブラリに依存せず、Node.js 標準モジュールのみで構成される自己完結的な設計である。
    - **`npm install` の禁止**: ルートディレクトリでの `npm install` は実行してはならない。
    - **標準モジュールの優先**: `scripts/` 内のツール（`todo.mjs`, `gemini_sub.mjs` 等）は `fs`, `path`, `child_process` 等の標準機能のみを使用している。依存関係を確認する際は、まずスクリプトの `import` 文を確認せよ。
    - **サブモジュールの隔離**: `superpowers-original/` 内の `package.json` は移植元の参照用であり、親プロジェクトの動作には関由しない。

- **ツールの使用とガイドの確認**:
    - `scripts/` 内のツール（`todo.mjs` 等）を使用する前には、必ずそのディレクトリに専用のガイド（例: `scripts/TODO_GUIDE.md`）が存在するかを確認し、存在する場合は必ず `read_file` で読解せよ。
    - 「以前使ったから知っている」という思い込み（Drifting）を排除し、現在の物理的な仕様（SSOT）に基づいて操作を開始せよ。

- **一時ファイルの混入防止 (Cleanup Discipline)**:
    - `replace`, `sed`, `grep` などのテキスト操作ツール、またはビルドツールを使用した直後は、`.!` や `.tmp` で始まる一時ファイルが生成されていないか `ls -a` 等で物理的に確認せよ。
    - 意図しない一時ファイルが発見された場合は、コミット前に直ちに削除せよ。

- **物理・論理の一貫した責任 (End-to-End Responsibility)**:
    - 同期やアップデートを伴うタスクにおいて、「物理的な取得（git fetch/pull/checkout）」と「論理的なマージ（ロジック抽出・反映）」を分離せず、AI がその玄関口から出口（DoD）まで一貫して責任を持つ設計にせよ。
    - ユーザーに git 操作などの物理的な準備を期待せず、必要であれば自律的に物理層の更新を提案・実行せよ。
