# 知見： Superpowers スキル移植とアップデート

## 概要
`obra/superpowers` からの移植およびアップデート追随における「物理的な整合性」を担保する。

- **スキル更新の物理的反映フロー**: スキルファイル（`SKILL.md` 等）の変更を反映させるには、以下のステップを物理的な手続きとして踏まなければならない。
    1.  **バージョニング**: `gemini-extension.json` のバージョンを自律的にインクリメントする。
    2.  **マージとプッシュ**: 変更を `main` ブランチにマージし、リモートへ `git push` する。（外部インストール環境への反映のため）
    3.  **拡張機能の即時更新（ローカル開発時）**: `gemini extensions reload`（または `gemini extensions update <extension-name>`）をユーザーに依頼することで、セッションを再起動することなく最新のスキル定義をロードできる。
    4.  **反映の物理的確認**: `activate_skill` を実行し、出力された instructions が最新（修正後の内容）であることを物理的に確認せよ。
- **バージョニングの義務化 (DoD)**:
    - スキルの新規作成、修正、または知見（Observations）の更新など、配布に影響する変更を加えた際は、ユーザーから指摘される前に自律的に `gemini-extension.json` のバージョンをインクリメントせよ。これを「完了の定義（DoD）」の最優先事項とする。
- **Atomic 0: 物理構成の同期規律**: `SKILL.md` の内容比較（差分分析）の前に、必ず `list_directory` を移植元とローカルの両方で実行し、ファイル構成の差分（増減）を物理的に特定せよ。
    - **新規ファイル**: 移植元にのみ存在するファイルは、メインセッションでの直接翻訳・移植、または（大規模な場合に限り）サブエージェントを検討し、独立したアトミック・タスクとして `todo.mjs` に登録せよ。
    - **削除ファイル**: 移植元で削除されたファイルは、ローカルでも不要かユーザーに確認の上、削除タスクとして登録せよ。
    - **孤児（Orphan）ファイルの回避**: `SKILL.md` 内で参照されている画像、スクリプト、補助ドキュメントが、物理的にディレクトリ内に存在するかをスキャンし、欠落している場合は「不備」として直ちに修正せよ。
- **移植元情報の正確な特定**:
    - アップデート追随（Upstream Sync）時、移植元の最新コミットハッシュは、サブモジュールディレクトリ内で `git log -1 --pretty=format:%H <path/to/skill>` 等を実行して物理的に特定せよ。推測や、サブモジュール全体の HEAD を鵜呑みにすることは、記録の不整合を招くため厳禁とする。

- **実証テストの義務化 (Step 4)**: スキルのアップデート（Upstream Sync）が完了したと主張する前に、必ず隔離されたワークツリー環境で **「実際にスキルを起動して、その出力が新しい指示に従っているか」** を物理的な証拠（ログ）をもって証明しなければならない。
- **不備の発見は「成功」である**: 実証テストやレビュー（サブエージェント等）で不備が発見された場合、それはスキルの改善が正しく機能している証拠である。直ちに修正して計画を洗練せよ。
- **マニュアル・レビューのフォールバック**: `code-reviewer` サブエージェントが不安定な場合は、単なる「動作しました」という報告ではなく、移植元と移植後のファイルを **「行単位で突き合わせた詳細な比較レビュー報告」** を自ら作成し、ユーザーに提示せよ。
- **補助プロンプトの活用**: スキル本体（`SKILL.md`）の指示だけでなく、補助プロンプト（`plan-document-reviewer-prompt.md` 等）がフローの中で正しく呼ばれるよう、スキル側に明示的な呼び出し手順を記述せよ。
