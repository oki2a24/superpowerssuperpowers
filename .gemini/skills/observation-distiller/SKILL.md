# 知見蒸留器 (Observation Distiller)

## 概要
セッション中の「失敗」「成功」「ユーザーの指摘」を、スキルや憲法を汚染することなく、外部ファイル（Observations）として永続化・構造化するための規律です。

## いつ使用するか
- セッションを終了（リフレッシュ）する直前。
- 重要な設計上の指摘や、規律違反の注意を受けたとき。
- 大きな実装フェーズが完了したとき。

## コアプロセス

### 1. 知見の抽出 (Distillation)
セッションログを分析し、以下のエンティティ（対象）ごとに知見を整理せよ。
- **憲法 (GEMINI.md)**: AI の根本的な行動原理や作法。
- **各スキル (スキル名.md)**: 特定のタスク手順や、プロジェクト固有の癖。

### 2. 保存先の決定 (Placement)
抽出した知見を、以下のディレクトリ構成と命名規則に従って仕分けよ。

- **ファイル名:** `{対象エンティティ名}.md` (例: `writing-plans.md`, `GEMINI.md`)
- **保存パス:**
    - **個人の流儀 (Global):** `~/.gemini/observations/`
    - **プロジェクトの掟 (Local):** `./.gemini/observations/`

### 3. 書き出しとマージ (Write & Merge)
対象の知見ファイルが既に存在する場合は、上書きではなく **「既存知見とのマージ」** を行え。重複を排除し、AI が即座にアクションに繋げられる「物理的な指示」へと抽象化せよ。
- **ディレクトリの自動作成:** 保存先ディレクトリが存在しない場合は、`run_shell_command("mkdir -p ...")` で作成せよ。
- **Git管理:** ローカル（`.gemini/observations/`）に保存する場合は、チーム全体で共有されるよう Git にコミットせよ。

## 知見の仕分けガイドライン

| 保存先 | 典型的な内容 | 理由 |
| :--- | :--- | :--- |
| **Global** | 「AI は常に RED ステップを省略してはならない」 | 普遍的な行動原理であり、どのプロジェクトでも有効。 |
| **Local** | 「このプロジェクトでは `scripts/todo.mjs` を優先せよ」 | プロジェクト固有のファイルパスやツール。 |
| **Local** | 「React コンポーネントは `src/components/ui/` に置く」 | プロジェクト固有の命名規則・構造。 |
| **Global** | 「`replace` ツールの `old_string` は多めに含める」 | ツールの使いこなしのコツ。 |

## 禁止事項
- **本体の汚染**: 知見を `GEMINI.md` や `SKILL.md` の本体（コード部分）に直接書き込んではならない。必ず `observations/` ディレクトリを使用せよ。
- **ログへの放置**: 重要な知見をチャットログの要約だけで終わらせてはならない。必ず物理的なファイルとして永続化せよ。

## よくある間違い
- **対象を特定しない**: どのスキルに対する知見か不明確な「雑記」として保存する。 -> 必ず対象名（スキル名等）と一致したファイル名にせよ。
