# Superpowersスキル移植ガイド

このスキルは、`obra/superpowers`からGemini CLIへのスキル移植プロセスを、以下の標準作業手順書（SOP）に従ってガイドします。

## 手動スキル移植の標準作業手順書(SOP)

### ステップ1: 準備とブランチ作成

1.  **移植対象スキルの決定:**
    *   **AIエージェントへの指示**: まず、移植対象のSuperpowersスキル名をユーザーに尋ねてください。（例: brainstorming, using-git-worktrees）

2.  **移植用ブランチの作成:**
    *   **AIエージェントへの指示**: `git checkout -b feat/port-<skill_name>` を実行して新しいブランチを作成してください。
    *   **理由**: スキル実装フェーズでは、パス指定（`dir_path`）の煩雑さを避け、実装プロセスをシンプルに保つため、メインのワークスペースで作業することを推奨します。

3.  **移植元リポジトリの確認:**
    *   **AIエージェントへの指示**: 移植元である `obra/superpowers` リポジリは、本リポジリの `superpowers-original` ディレクトリにサブモジュールとして追加されています。サブモジュールのパスは `./superpowers-original` です。AIエージェントは常にこのパスを基準としてファイルを読み込むようにしてください。

### ステップ2: 元のSKILL.mdの読み込みと理解

*   決定されたスキル名と`superpowers-original`のパスを使って、移植対象となる`{superpowers_original_path}/skills/{skill_name}/SKILL.md`を読み込みます。
*   読み込んだ内容から、スキルの目的、ワークフロー、意図を詳細に理解します。理解が完了したら、その概要をユーザーに報告し、次のステップに進む確認してください。
*   **AIエージェントへの指示**: ワークツリーを使用している場合、これ以降のツール呼び出し（`read_file`, `write_file`, `run_shell_command` など）では、必ず `dir_path` パラメータにワークツリーのパスを指定して実行してください。

### ステップ3: Gemini CLIへの適応分析

*   スキルの核となる目的とパラメータ、トリガー条件を特定します。
*   元のSKILL.md内でClaude固有の記述をリストアップします。
    *   `/plugin`コマンド、`Task("...")`、`CLAUDE.md`への参照
    *   `digraph`ブロック（**重要: これらは必ずMermaid形式に変換して記述します**）
    *   `superpowers:`プレフィックスのスキル呼び出し
*   これらのClaude固有の要素を、Gemini CLIの標準ツール（`run_shell_command`、`write_file`、対話など）や本プロジェクト独自のタスク管理スクリプト（`scripts/todo.py`）でどのように代替または表現するかを検討し、具体案を立案します。
*   分析結果と代替案をユーザーに提示し、必要に応じて追加の質問を行ってください。ユーザーの承認を得てから次のステップに進んでください。

### ステップ4: Gemini CLI用SKILL.mdのドラフト作成

*   上記の分析に基づき、Gemini CLI形式の新しい`SKILL.md`のコンテンツをドラフトします。
*   **重要:** 移植するスキルの内容は、このプロジェクトの他のスキルと一貫性を保つため、**日本語に翻訳・統一**してください。
*   （コンテンツ構造の原則は変更なし）
*   ドラフトが完成したら、ユーザーに提示し、レビューと承認を求めてください。
*   **デフォルトスキルとの重複チェックと対応:**
    *   **AIエージェントへの指示**: `gemini skills list`を実行し、組み込みスキルの中にドラフトされたスキル名と同じ名前のものが存在するかを確認します。
    *   もし存在する場合、ユーザーにその旨を報告し、以下の選択肢を提示します。
        *   そのままのスキル名で上書きを許容するか。
        *   スキル名を変更して共存させるか（例: `<skill_name>-custom` や `my-<skill_name>`）。
    *   ユーザーの選択に基づいて、必要であればスキル名を修正し、続行します。

### ステップ5: 新規SKILL.mdの作成とコミット

*   ユーザーの承認を得たSKILL.mdのドラフトを、`./.gemini/skills/{skill_name}/SKILL.md`として`write_file`ツールで書き込みます。
*   Gitに`git add`し、コミットメッセージ「feat: <skill_name>スキルを移植する (SKILL.md)」で現在のブランチにコミットします。

### ステップ6: 移植情報の記録

*   **AIエージェントへの指示**: 実装コミット後、速やかに `update-superpowers-ports-doc` スキルを起動し、`docs/superpowers_ports.md` に移植情報を記録してください。
*   **理由**: 移植作業が完了した熱量があるうちに記録を行うことで、ドキュメントの更新漏れを防ぎ、履歴の一貫性を保ちます。

### ステップ7: 実装完了とセッションのリフレッシュ

*   **AIエージェントへの指示**: 実装したスキルをGemini CLIに認識させ、テスト環境を構築するために、現在のセッションを一度終了（リフレッシュ）することをユーザーに提案してください。
*   **再開用プロンプトの作成**: 次の「ステップ8：テスト環境の構築」をスムーズに行うため、以下の内容を含む再開用プロンプトを作成します。
    *   移植したスキルの概要
    *   テスト用ブランチ名（例: `test/run-<skill_name>-test`）の提案
    *   テスト用ワークツリーの作成ディレクトリ案

### ステップ8: テスト環境の構築と実行 (新しいセッション)

1.  **テスト環境の構築:**
    *   **AIエージェントへの指示**: `GEMINI.md`の原則に従い、テストは隔離された環境で行います。
    *   **AIエージェントへの指示**: `using-git-worktrees`スキルを有効化し、`test/run-<skill_name>-test` という名前の新しいブランチとワークツリーを作成して、テストを実行します。
    *   **重要 (スキルの認識)**: テスト用ワークツリーのディレクトリへ移動し、そこでGemini CLIを**再起動**（`gemini --resume latest` を実行）してください。これにより、新しく作成したスキルを確実に認識させることができます。
    *   **注意**: この「再起動」は現在のコンテキスト（最新のターン）を維持するためのものであり、タスク中断時の「セッションのリフレッシュ（再開用プロンプトの作成）」とは異なります。

2.  **テストの実行:**
    *   新しく作成されたテスト用ワークツリーに移動します。
    *   **AIエージェントへの指示**: ユーザーにテストの実行方法を尋ねてください。
        *   **A) 手動対話モード:** ユーザーがシナリオを提供し、AIからの質問に直接回答するモード。
        *   **B) AI主導の自己完結テストモード:** AIがテストシナリオを自律的に設計し、自己対話形式で実行するモード。
    *   選択されたモードでテストを実行します。

### ステップ9: テスト環境のクリーンアップ

*   **AIエージェントへの指示**: テスト完了後、`finishing-a-development-branch`スキルを起動し、「この作業を破棄する」オプションを選択することで、テスト用のワークツリーとブランチを安全にクリーンアップできることをユーザーに伝えて実行します。

### ステップ10: フィーチャーブランチのマージ

*   **AIエージェントへの指示**: テスト環境のクリーンアップ後、`finishing-a-development-branch`スキルを再度起動します。
*   **AIエージェントへの指示**: ユーザーに、移植したスキルを含むフィーチャーブランチ（`feat/port-<skill_name>`）を`main`ブランチにマージすることを提案し、「1. ローカ ルで<ベースブランチ>にマージする」オプションを選択して実行します。

### 既存スキルのアップデート手順（ローカル・アダプテーション保護）

移植済みのスキルが `obra/superpowers` 側でアップデートされた場合、ローカルで行った改善（`## ローカル・アダプテーション (Gemini固有)` セクション）が失われないように、以下の手順でアップデートを適用します。

1.  **既存のローカル・アダプテーションのバックアップ:**
    *   **AIエージェントへの指示**: `cat ./.gemini/skills/{skill_name}/SKILL.md` を実行し、既存の `SKILL.md` の内容を取得します。
    *   取得した内容から、`## ローカル・アダプテーション (Gemini固有)` セクション（ヘッダーとそれ以降の全行）を抽出し、一時ファイル（例: `/tmp/{skill_name}_local_adaptation.md`）に保存します。

2.  **元のSuperpowers版の最新内容を適用:**
    *   **AIエージェントへの指示**: `superpowers-original` リポジトリを最新の状態に更新し、アップデートされた `{superpowers_original_path}/skills/{skill_name}/SKILL.md` の内容を取得します。
    *   **AIエージェントへの指示**: 取得した最新の内容を、`./.gemini/skills/{skill_name}/SKILL.md` に `write_file` で上書きします。

3.  **ローカル・アダプテーションの再結合:**
    *   **AIエージェントへの指示**: ステップ 1 でバックアップしたローカル・アダプテーションの内容を、ステップ 2 で上書きした `SKILL.md` の末尾に追記します。
    *   **AIエージェントへの指示**: `git diff` を実行し、変更内容を確認します。

4.  **動作検証とコミット:**
    *   **AIエージェントへの指示**: 更新されたスキルに対してテストを実行し、意図通りに動作することを確認します。
    *   **AIエージェントへの指示**: `git add ./.gemini/skills/{skill_name}/SKILL.md` を実行し、コミットメッセージ「fix: <skill_name>スキルをアップデートする（ローカル変更保護済み）」でコミットします。

### ステップ11: 完了と次のステップ

1.  **SOPの継続的な改善:**
    *   移植作業を通じて得られた新しい知見や改善点は、このSOPにフィードバックし、常に最新かつ最適な手順となるように更新します。

2.  **セッションのリフレッシュ提案:**
    *   **AIエージェントへの指示**: 「スキル移植が完了しました。コンテキストが大きくなったため、セッションをリフレッシュしますか？」とユーザーに尋ねてください。
    *   **はいの場合:**
        *   **AIエージェントへの指示**: 次のセッションで作業をスムーズに再開するための要約と次のタスク提案を含む**再開用プロンプト**を生成し、ユーザーに提示してください。
        *   例: `「<skill_name>スキルの移植が完了し、ブランチ <feat/port-skill_name> にコミットされています。次のタスクとして、<次のスキル>の移植、またはプロジェクト 基盤の整備（README作成など）が考えられます。どちらから着手しますか？」`
    *   **いいえの場合:**
        *   **AIエージェントへの指示**: ユーザーに次のスキル移植に進むか、別のタスクに進むかを確認してください。
