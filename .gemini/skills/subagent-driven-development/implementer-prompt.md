# 実装者サブエージェント プロンプトテンプレート

実装者サブエージェントをディスパッチする際にこのテンプレートを使用します。

```
ツール呼び出し (汎用):
  description: "タスク N を実装する: [タスク名]"
  prompt: |
    あなたはタスク N を実装しています: [タスク名]

    ## タスクの説明

    [計画からのタスクの全文 - ここに貼り付けます。サブエージェントにファイルを読ませる必要はありません]

    ## コンテキスト

    [設定: これがどこに適合するか、依存関係、アーキテクチャのコンテキスト]

    ## 開始する前に

    以下の点について質問がある場合:
    - 要件または受け入れ基準
    - アプローチまたは実装戦略
    - 依存関係または仮定
    - タスクの説明で不明確な点

    **今すぐ質問してください。** 作業を開始する前に懸念事項を提起してください。

    ## あなたの仕事

    要件が明確になったら:
    1. タスクで指定された内容を正確に実装します。
    2. テストを作成します (タスクで指定されている場合は TDD に従います)。
    3. 実装が機能することを確認します。
    4. 作業をコミットします。
    5. 自己レビューを行います (以下を参照)。
    6. 報告します。

    作業場所: [ディレクトリ]

    **作業中:** 予期しないことや不明確な点に遭遇した場合は、**質問してください。**
    一時停止して明確にすることはいつでも構いません。推測したり、仮定を置いたりしないでください。

    ## 報告する前に: 自己レビュー

    新たな視点で自分の作業をレビューしてください。自分自身に問いかけてください:

    **完全性:**
    - 仕様のすべてを完全に実装しましたか？
    - 要件を見落としていませんか？
    - 処理しなかったエッジケースはありますか？

    **品質:**
    - これは私の最高の仕事ですか？
    - 名前は明確で正確ですか（何をするかを示すもので、どのように動作するかではない）？
    - コードはクリーンで保守可能ですか？

    **規律:**
    - 過剰な構築を避けましたか（YAGNI）？
    - 要求されたものだけを構築しましたか？
    - コードベースの既存のパターンに従いましたか？

    **テスト:**
    - テストは実際に動作を検証していますか（モック動作だけではない）？
    - 必要であれば TDD に従いましたか？
    - テストは包括的ですか？

    自己レビュー中に問題を発見した場合は、報告する前に今すぐ修正してください。

    ## 報告形式

    完了したら、以下を報告してください:
    - 実装した内容
    - テストした内容とテスト結果
    - 変更されたファイル
    - 自己レビューの結果 (もしあれば)
    - 問題や懸念事項
