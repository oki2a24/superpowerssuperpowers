# Systematic Debugging

## Overview

このスキルは、バグ、テスト失敗、予期せぬ動作などの技術的な問題に直面した際に、場当たり的な修正を避け、根本原因を体系的に特定するための厳格なプロセスをAIエージェントに強制します。

**中核となる原則:** 修正を試みる前に、必ず根本原因の調査を完了させること。

**このプロセスの方針に違反することは、デバッグの精神そのものに違反する行為とみなされます。**

## The Process

### ステップ1: 起動とTODOリストの作成

1.  このスキルが起動されたら、まず「体系的デバッグプロセスを開始します。根本原因の特定を最優先します。」と宣言します。
2.  次に、`run_shell_command`を使い、`python3 scripts/todo.py init "Systematic Debugging"` でTODOファイルを初期化した後、以下の4つのフェーズを `python3 scripts/todo.py add` で追加してください。以降の作業は、このTODOリストに従って厳格に進行し、各ステップの開始時に `python3 scripts/todo.py start <パターン>` を、完了時に `python3 scripts/todo.py done` を実行してステータスを更新してください。

    *   `Phase 1: 根本原因の調査`
    *   `Phase 2: パターン分析`
    *   `Phase 3: 仮説とテスト`
    *   `Phase 4: 実装`

### ステップ2: Phase 1: 根本原因の調査 (Root Cause Investigation)

**目的:** コードの変更を一切行わず、問題が「何」であり、「なぜ」発生するのかを完全に理解する。

1.  **エラーメッセージとスタックトレースの精読:** `read_file`やログ確認を通じて、エラーメッセージの全ての行を注意深く読み、ファイルパス、行番号、エラーコードを正確に把握します。
2.  **一貫した再現手順の確立:** 問題を確実に再現できる手順を特定します。再現ができない場合は、推測で進めず、さらなる情報収集（ログの追加など）を行います。
3.  **最近の変更の確認:** `git log -p -5`や`git diff`などの`run_shell_command`を利用して、問題に関連しそうな最近のコード変更、設定変更、依存関係の更新を調査します。
4.  **証拠の収集:** 複数のコンポーネントが絡むシステムの場合、各コンポーネントの境界で何が起きているかを可視化します。`run_shell_command`を使い、データがコンポーネントに入出力される箇所に一時的なログ（例: `echo`）を追加するコマンドを実行し、問題がどのレイヤーで発生するかを特定します。
5.  **データフローの追跡:** `grep_search`や`read_file`を駆使し、不正な値や状態がどこから来たのかを、コールスタックを遡って追跡します。

**このフェーズが完了するまで、次のフェーズに進んではなりません。**

### ステップ3: Phase 2: パターン分析 (Pattern Analysis)

1.  コードベース内から、現在問題となっている機能と類似しているが、**正常に動作しているコード**を探します。
2.  正常なコードと問題のコードを比較し、たとえ些細であっても、すべての差異をリストアップします。
3.  そのコードが依存している他のコンポーネント、設定、環境変数を理解します。

### ステップ4: Phase 3: 仮説とテスト (Hypothesis and Testing)

1.  調査結果に基づき、「**Xが根本原因である。なぜならYだからだ。**」という形式で、単一の明確な仮説をユーザーに提示します。
2.  その仮説を検証するための、**最小限の変更**を提案します。一度に複数の変数を変更してはいけません。
3.  変更によって仮説が証明されたか（問題が解消したか）を確認します。もし解決しなかった場合は、**変更を元に戻し**、新たな情報を元に新しい仮説を立てます（ステップ4-1に戻る）。

### ステップ5: Phase 4: 実装 (Implementation)

1.  **失敗するテストケースの作成:**
    *   修正に着手する前に、問題を再現する最もシンプルなテストケースを作成する必要があります。
    *   `activate_skill('test-driven-development')` を使用して、適切な失敗するテストを作成するようユーザーに提案し、その指示に従ってください。
2.  **単一の修正の実装:**
    *   特定された根本原因を直接解決する、単一の修正を実装します。
    *   「ついでに」のリファクタリングや改善を行ってはいけません。
3.  **修正の検証:**
    *   作成したテストが成功すること、そして他の既存テストが失敗していないことを確認します。
4.  **3回以上の修正が失敗した場合の対処:**
    *   もし、3回試みても問題が解決しない場合、それは**アーキテクチャ上の問題**を示唆しています。
    *   **直ちに修正の試みを中止**し、「3回の修正を試みましたが、根本原因はより深いアーキテクチャ上の問題である可能性が高いです。アーキテクチャの基本設計について再検討しませんか？」とユーザーに問いかけ、議論を促してください。**4回目の修正を試みてはなりません。**

## 参考情報

### Red Flags - この思考に陥ったらプロセスに戻れ

*   「とりあえずの修正で、調査は後回し」
*   「Xを変更してみて、うまくいくか見てみよう」
*   「テストは省略して、手動で確認する」
*   「問題はたぶんXだろうから、そこを直す」
*   「すでに2回以上修正を試みたが、もう一回だけ試してみよう」

### ユーザーからの危険信号

ユーザーが以下のような指摘をした場合、あなたの進め方が間違っている可能性があります。直ちにPhase 1に戻ってください。
*   「それは確認しなくてもいいの？」 (確認を怠り、仮定で進めている)
*   「推測はやめて」 (証拠に基づかずに修正案を提示している)
