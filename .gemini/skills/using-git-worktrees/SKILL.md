# Gitワークツリーの利用

## 概要

Gitワークツリーを使用して隔離されたワークスペースをセットアップします。
Gitワークツリーは、同じリポジトリを共有しながらも隔離された作業環境を作成し、ブランチを切り替えることなく複数のブランチで同時に作業することを可能にします。

**コア原則:** 系統的なディレクトリ選択と安全性検証により、信頼性の高い隔離を実現します。

## プロセス

### 1. ディレクトリ選択プロセス

以下の優先順位に従って、ワークツリーを作成するディレクトリを決定します。

*   **既存ディレクトリの確認**:
    *   現在のプロジェクトディレクトリ内に`.worktrees/`または`worktrees/`ディレクトリが存在するかを`ls -d .worktrees 2>/dev/null`および`ls -d worktrees 2>/dev/null`コマンドで確認します。
    *   両方存在する場合は`.worktrees/`を優先します。
    *   見つかった場合は、そのディレクトリを使用します。

*   **`GEMINI.md` (または設定ファイル) の確認**:
    *   `.gemini/settings.json`など、プロジェクト固有の設定ファイルにワークツリーディレクトリのパスが指定されていないか確認します。指定されている場合はそれを使用します。

*   **ユーザーへの確認**:
    *   既存のワークツリーディレクトリが見つからず、設定ファイルにも指定がない場合、ユーザーにワークツリーを作成する場所を尋ねます。
    *   選択肢として以下を提示します。
        1.  `.worktrees/` (プロジェクトローカル、非表示)
        2.  `~/.gemini/worktrees/<project-name>/` (グローバルロケーション - `<project-name>`は現在のリポジトリ名)
    *   ユーザーにどちらを希望するかを尋ね、選択されたパスを使用します。

### 2. 安全性検証

ワークツリーを作成する場所が決定されたら、以下の安全性検証を行います。

*   **プロジェクトローカルディレクトリの場合 (`.worktrees/` または `worktrees/`)**:
    *   **重要**: ディレクトリがGitによって無視されていることを必ず検証します。
    *   `git check-ignore -q .worktrees 2>/dev/null || git check-ignore -q worktrees 2>/dev/null`コマンドを実行し、ディレクトリが`.gitignore`で無視されているかを確認します。
    *   **無視されていない場合**:
        *   適切な行を`.gitignore`ファイルに追加し、コミットします。
        *   **理由**: ワークツリーの内容が誤ってリポジトリにコミットされるのを防ぐため、これは非常に重要です。
*   **グローバルディレクトリの場合 (`~/.gemini/worktrees/`)**:
    *   プロジェクトの外に作成されるため、`.gitignore`による検証は不要です。

### 3. 作成ステップ

*   **プロジェクト名の検出**: `project=$(basename "$(git rev-parse --show-toplevel)")`コマンドを実行し、現在のGitリポジトリのプロジェクト名を検出します。
*   **ワークツリーの作成**:
    *   決定した`LOCATION`と検出した`project`名、ユーザーが指定した`branch_name` (`parameters.branch_name`) を使用して、ワークツリーのフルパスを決定します。
    *   `git worktree add "$path" -b "$branch_name"`コマンドを実行して、新しいブランチでワークツリーを作成します。
    *   `cd "$path"`コマンドを実行して、作成されたワークツリーディレクトリに移動します。
*   **プロジェクトセットアップの実行**:
    *   ワークツリーディレクトリ内で、`package.json`, `Cargo.toml`, `requirements.txt`, `pyproject.toml`, `go.mod`のいずれかのファイルをチェックし、適切な依存関係のインストールコマンドを自動検出して実行します。
*   **クリーンベースラインの検証**:
    *   プロジェクトに適したテストコマンド（例: `npm test`、`cargo test`、`pytest`、`go test ./...`）を実行し、ワークツリーがクリーンな状態で開始されたことを確認します。
    *   **テストが失敗した場合**: 失敗をユーザーに報告し、続行するか、それとも調査するかを尋ねます。
    *   **テストが成功した場合**: 準備完了であることを報告します。
*   **場所の報告**:
    *   ワークツリーが準備できたこと、テスト結果、そして`branch_name`を含む機能名と共に、ワークツリーのフルパスをユーザーに報告します。

## 参考情報

### クイックリファレンス

| 状況 | アクション |
|-----------|--------|
| `.worktrees/` が存在する場合 | それを使用する (無視されていることを検証) |
| `worktrees/` が存在する場合 | それを使用する (無視されていることを検証) |
| 両方存在する場合 | `.worktrees/` を使用する |
| どちらも存在しない場合 | `GEMINI.md` (または設定ファイル) を確認 → ユーザーに尋ねる |
| ディレクトリが無視されていない場合 | `.gitignore` に追加 + コミット |
| ベースラインテスト中にテストが失敗した場合 | 失敗を報告 + ユーザーに尋ねる |
| `package.json`/`Cargo.toml` がない場合 | 依存関係のインストールをスキップ |

### よくある間違い

*   **無視検証のスキップ**:
    *   **問題:** ワークツリーの内容が追跡され、Gitの状態を汚染する
    *   **修正:** プロジェクトローカルなワークツリーを作成する前に、常に`git check-ignore`を使用する
*   **ディレクトリリセットの失念**:
    *   **問題:** `run_shell_command` の `cd` はその呼び出し内でのみ有効であり、次のツール呼び出しではルートに戻ってしまう。
    *   **修正:** ワークツリー内で継続的な作業を行う場合は、各ツール呼び出しの `dir_path` パラメータにワークツリーのパスを明示的に指定する。
*   **ディレクトリの場所の仮定**:
    *   **問題:** 一貫性がなくなり、プロジェクトの慣習に反する
    *   **修正:** 優先順位に従う: 既存 > `GEMINI.md` (または設定ファイル) > 尋ねる
*   **失敗したテストで続行**:
    *   **問題:** 新しいバグと既存の問題を区別できない
    *   **修正:** 失敗を報告し、続行するための明示的な許可を得る
*   **セットアップコマンドのハードコーディング**:
    *   **問題:** 異なるツールを使用するプロジェクトでは動作しない
    *   **修正:** プロジェクトファイル (`package.json`など) から自動検出する

### 危険な兆候

**決してしないこと:**
*   無視されていることを検証せずにワークツリーを作成する (プロジェクトローカル)
*   ベースラインテストの検証をスキップする
*   テストが失敗したまま、尋ねずに続行する
*   曖昧な場合にディレクトリの場所を仮定する
*   `GEMINI.md` (または設定ファイル) の確認をスキップする

**常にすること:**
*   優先順位に従う: 既存 > `GEMINI.md` (または設定ファイル) > 尋ねる
*   プロジェクトローカルなディレクトリが無視されていることを検証する
*   プロジェクトのセットアップを自動検出して実行する
*   クリーンなテストベースラインを検証する

### 統合

*   **呼び出し元**: `brainstorming` (フェーズ4) - 設計承認後に実装が続く場合に必須, `subagent-driven-development` - タスク実行前に必須, `executing-plans` - タスク実行前に必須, 孤立したワークスペースを必要とするあらゆるスキル
*   **連携先**: `finishing-a-development-branch` - 作業完了後のクリーンアップに必須
