# Gemini CLI 運用記憶と学習記録 (SSOT)

本ファイルは、Gemini CLI エージェントがこのプロジェクトにおいて遵守すべき行動規範、技術的決定事項、および実戦から得た知見を永続的に記録した唯一の真実のソース（SSOT）である。

## 1. AIエージェントの行動規律 (Core Mandates)

### Git 操作の徹底 (Git Discipline)
- **ブランチの保護**: 開発着手前、および**コミット直前**に必ず現在のブランチを確認すること。
    - **物理的確認**: `git branch --show-current` を実行し、ブランチ名を確認すること。
    - **警告と中断**: もし現在のブランチが `main` (または `master`) であった場合、直ちに **「【警告】現在のブランチは main です。変更をコミットしてはいけません。」** という一文を出力し、作業ブランチの作成または移動を行え。この警告を出力したターンにおいて、**いかなる理由があっても（例：小さなドキュメント修正であっても）コミットを強行してはならない。** 必ず次ターンでブランチ移動のステップを挟め。
- **原子的なコミット (Atomic Commits)**: TDD の各タスク（GREEN 達成時）ごとに、その成果を即座にコミットすること。コミット漏れは履歴の劣化を招くため、タスク完了の必須条件とする。

### サブセッション運用と統合 (GPAC v3)
- **起動の鉄則**: `gemini-sub spawn` は原則として対象のワークツリー内で実行せよ。その際、プロジェクト名の一貫性を保つため `-p <project_name>` オプションを明示すること。
- **統合プロトコルの完結**: `import` コマンドによる報告確認後、直ちに成果ブランチのメインブランチへのマージと、不要になったワークツリーの `--force` 削除（未追跡ファイルのクリーンアップを含む）を提案し、作業環境を原子的に統合せよ。

### TDD とドキュメントの融合
- **設計としてのコメント**: テストケースの Docstring およびテストタイトル（第1引数、日本語）を記述することを、TDD の仕様策定（Red フェーズ）の必須ステップとせよ。日本語で意図を言語化することが、正確な実装への最短ルートとなる。
- **図解による構造化**: テストコード内には、アスキーアート等を用いた [イメージ] 図解を積極的に残せ。特に階層構造や状態遷移を扱う場合、図解は「唯一の真実の仕様」として保守性を飛躍的に高める。
- **試行錯誤の継承**: 実装時のトリッキーなロジックや解決済みの曖昧さについては、将来の自己改善のためにその背景（Context）を必ずコード内に残せ。

### タスク階層管理の仕様 (todo.mjs)
- **最深部優先の法則**: `add --child` や `done` は、常に「現在実行中 [/] のタスクの中で最もインデントが深いもの」を優先的に対象とせよ。これにより、多階層（3レベル以上）における親タスクの誤認を防止する。


### 進捗管理と同期
- **ロードマップの提示**: 複数ステップにわたる大規模タスクでは、開始時に「ステップごとの使用スキル」を含むロードマップを提示すること。
- **現在地の明示 (Full Reporting)**: 各タスクの変わり目（コミット直後など）には、必ず「現在地：Task M 完了 / 次：Task M+1」を報告すること。その際、**次タスクの「名称」「使用予定スキル」「GREEN条件」を一切省略せずに再掲せよ**。情報の圧縮によるユーザーの不安を排除し、常に次に適用される規律を相互確認すること。

## 2. 開発・テストワークフロー

### 作業の分離の原則
- **フィーチャー開発**: 永続的な変更は必ず専用のフィーチャーブランチで行う。
- **検証環境の隔離**: 実装環境を汚染せず、スキルの再起動等を確実に行うため、テストや検証は `using-git-worktrees` を使用した独立したワークツリーで実行することを原則とする。

### 技術選定と低オーバーヘッド
- **標準ライブラリの優先**: 環境構築のコストを最小化しポータビリティを高めるため、Python および Node.js においては標準ライブラリのみで完結させる実装を最優先する。外部パッケージ（pip/npm等）への依存は、その必要性を厳格に評価した上で導入すること。

### セッション管理の定義
- **セッション・リフレッシュ**: 履歴を破棄してコンテキストを節約する行為。再開用プロンプトを必須とする。
- **スキルの認識（再起動）**: `gemini --resume latest` を使い、履歴を保持したまま新しいスキルを読み込ませる行為。

## 3. セッション間連携 (GPAC)

### プロトコルの遵守
- **SSOT の場所**: サブセッションのデータは `~/.gemini/sub-sessions/` に集約し、プロジェクト横断的な一貫性を維持する。
- **Workspace Boundary の克服**: ワークスペース外（境界外）のディレクトリ操作には、`cat`/`cp` 等の低レイヤーなシェルコマンドではなく、`gemini-sub` が提供する専用 API (`show-task`, `show-report` 等) を優先して使用すること。
- **Handoff (引き渡し) 方式**: 書き込み制限を回避するため、ワークスペース内で「下書き」を仕上げ、スクリプトがそれを検証・正規化してグローバル領域へ移動させるフローを標準とする。

### テストとデグレードの防止
- **ベースラインの尊重**: 既存テストが存在するファイルを修正・刷新する場合、**既存のテストコードを削除してはならない**。必ず既存のテストスイートをベースラインとし、TDD のサイクルごとに一つずつ新仕様へ適合させること。全文書き換えによるショートカットは禁止する。
- **全件テストの原則**: 特定のタスクに関連するテストだけでなく、原則としてプロジェクト内の**全件テスト（例: `node --test tests/test_*.mjs`）を実行せよ**。これにより、修正対象外のファイルで発生した構文エラーや予期せぬ副作用を早期に検知すること。
- **物理的検証の優先**: `mock` による仮想的な検証よりも、`tmpdir` 等を用いた物理的なファイル I/O 検証を優先せよ。

### ツール使用の安全策
- **一括書き込みの制限**: 50行を超える既存ファイルへの変更、または複雑なエスケープ（改行、クォート）を含む変更には、`write_file` ではなく `replace` ツールを使用せよ。一括書き込みは `SyntaxError` を招くリスクが高いため、原則として新規作成時のみに限定する。
