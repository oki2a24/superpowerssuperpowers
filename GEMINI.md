## Gemini CLI エージェントの運用記憶と学習記録

このファイルは、Gemini CLIエージェントがこのプロジェクトのコンテキストで得た知見、作業のベストプラクティス、および特定のルールを永続的に記録するためのものです。エージェントは、この情報を参照し、将来のタスク実行に役立てます。

### 1. スキルテスト実施手順 (2026年2月10日)

新しいスキルを開発およびテストする際の標準的な手順を以下に示します。これは、実装の成果物とテストによる一時的な成果物を明確に分離し、プロジェクトのクリーンな状態を維持することを目的とします。

1.  **実装とテストの分離:**
    *   **実装フェーズ**:
        1.  新しいスキルごとに、専用のGitブランチを作成します (例: `feat/add-<skill-name>-skill`)。
        2.  スキルのディレクトリ構造と`SKILL.md`ファイルを作成します。
        3.  これらの**実装ファイルのみを**、明確なコミットメッセージと共にブランチにコミットします。これらは恒久的な成果物とみなします。
    *   **テストフェーズ**:
        1.  実装が完了した同じブランチ上で、`activate_skill`ツールを使い、テストを開始します。

2.  **テスト成果物の事前特定:**
    *   スキルを実行する**前に**、そのスキルが一時的に生成すると予想される成果物（ファイル、ディレクトリ、Gitブランチ、コミットなど）を明示的にリストアップします。これらは「テストアーティファクト」と呼びます。

3.  **テストの実行:**
    *   `activate_skill`を実行し、その動作と出力を観察します。

4.  **テストアーティファクトのクリーンアップ:**
    *   テスト完了後、ステップ2で特定した**「テストアーティファクト」のみを**クリーンアップ（削除・元に戻す）します。
    *   **重要:** ステップ1でコミットした**スキル実装ファイル（`SKILL.md`など）は、クリーンアップの対象外**とし、ブランチに永続的に残します。
    *   クリーンアップには、`git worktree remove`、`git reset HEAD~1`（一時的なコミット用）、`git restore`、`rm -rf`などの適切なコマンドを使用します。

5.  **結果の報告:**
    *   クリーンアップ完了後、「テストアーティファクトのクリーンアップが完了しました。実装されたスキルファイルは現在のフィーチャーブランチ上に残っています。」と明確に報告します。

### 2. Gitワークフローに関する方針 (2026年2月10日)

フィーチャー開発が完了し、テストとレビューを経て安定した状態になったブランチは、以下の手順でメインブランチに統合し、整理します。

1.  **フィーチャーブランチからメインブランチへのマージ**:
    *   開発が完了したフィーチャーブランチ (`feat/*` など) を、`main`ブランチ（またはそれに相当する安定版ブランチ）にマージします。
    *   マージは、通常`git merge --no-ff`またはGitHub等のプルリクエストのマージオプションを通じて行われ、マージコミットを作成します。
2.  **フィーチャーブランチの削除**:
    *   ローカルおよびリモートのリポジトリから、マージ済みのフィーチャーブランチを削除します。
        *   `git branch -d <feature-branch-name>`
        *   `git push origin --delete <feature-branch-name>`
3.  **新しいタスク用のブランチ作成**:
    *   次の開発作業を開始する前に、必ず`main`ブランチから新しいフィーチャーブランチを作成します。
        *   `git checkout main`
        *   `git pull`
        *   `git checkout -b feat/new-task-name`

`obra/superpowers`のスキルをGemini CLIに移植する際は、標準化された手順に従い、`port-superpowers-skill`メタスキルを使用します。このメタスキルは、移植作業の詳細な手順（SOP）を提供し、エージェント（私）の作業をガイドします。

### 4. 記憶ファイル (`GEMINI.md`) の運用方針

*   **継続的な更新**: このファイルは、プロジェクトの進行に伴う新たな学び、最適なプラクティス、重要な決定事項、およびエージェントの行動に関する指針を継続的に追記・更新します。
*   **構造化**: 各項目は明確な見出しを設け、日付を付与することで、参照しやすく、履歴を追えるようにします。
*   **意思決定の記録**: 特定の選択肢やアプローチを採用した理由、トレードオフに関する情報なども記録し、将来のコード理解や意思決定に役立てます。
*   **透明性**: エージェントの行動原理や思考プロセスを外部から理解できる形で記述します。
