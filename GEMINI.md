## Gemini CLI エージェントの運用記憶と学習記録

このファイルは、Gemini CLIエージェントがこのプロジェクトのコンテキストで得た知見、作業のベストプラクティス、および特定のルールを永続的に記録するためのものです。エージェントは、この情報を参照し、将来のタスク実行に役立てます。

### 1. スキルテスト実施手順 (2026年2月10日)

新しいスキルを開発およびテストする際の標準的な手順を以下に示します。これは、実装の成果物とテストによる一時的な成果物を明確に分離し、プロジェクトのクリーンな状態を維持することを目的とします。

1.  **実装とテストの分離:**
    *   **実装フェーズ**:
        1.  新しいスキルごとに、専用のGitブランチを作成します (例: `feat/add-<skill-name>-skill`)。
        2.  スキルのディレクトリ構造と`SKILL.md`ファイルを作成します。
        3.  これらの**実装ファイルのみを**、明確なコミットメッセージと共にブランチにコミットします。これらは恒久的な成果物とみなします。
    *   **テストフェーズ**:
        1.  実装が完了した同じブランチ上で、`activate_skill`ツールを使い、テストを開始します。

2.  **テスト成果物の事前特定:**
    *   スキルを実行する**前に**、そのスキルが一時的に生成すると予想される成果物（ファイル、ディレクトリ、Gitブランチ、コミットなど）を明示的にリストアップします。これらは「テストアーティファクト」と呼びます。

3.  **テストの実行:**
    *   `activate_skill`を実行し、その動作と出力を観察します。

4.  **テストアーティファクトのクリーンアップ:**
    *   テスト完了後、ステップ2で特定した**「テストアーティファクト」のみを**クリーンアップ（削除・元に戻す）します。
    *   **重要:** ステップ1でコミットした**スキル実装ファイル（`SKILL.md`など）は、クリーンアップの対象外**とし、ブランチに永続的に残します。
    *   クリーンアップには、`git worktree remove`、`git reset HEAD~1`（一時的なコミット用）、`git restore`、`rm -rf`などの適切なコマンドを使用します。

5.  **結果の報告:**
    *   クリーンアップ完了後、「テストアーティファクトのクリーンアップが完了しました。実装されたスキルファイルは現在のフィーチャーブランチ上に残っています。」と明確に報告します。

### 2. Gitワークフローに関する方針 (2026年2月10日)

フィーチャー開発が完了し、テストとレビューを経て安定した状態になったブランチは、以下の手順でメインブランチに統合し、整理します。

1.  **フィーチャーブランチからメインブランチへのマージ**:
    *   開発が完了したフィーチャーブランチ (`feat/*` など) を、`main`ブランチ（またはそれに相当する安定版ブランチ）にマージします。
    *   マージは、通常`git merge --no-ff`またはGitHub等のプルリクエストのマージオプションを通じて行われ、マージコミットを作成します。
2.  **フィーチャーブランチの削除**:
    *   ローカルおよびリモートのリポジトリから、マージ済みのフィーチャーブランチを削除します。
        *   `git branch -d <feature-branch-name>`
        *   `git push origin --delete <feature-branch-name>`
3.  **新しいタスク用のブランチ作成**:
    *   次の開発作業を開始する前に、必ず`main`ブランチから新しいフィーチャーブランチを作成します。
        *   `git checkout main`
        *   `git pull`
        *   `git checkout -b feat/new-task-name`

### 4. AIエージェントの行動原則と学び (2026年2月11日)

AIエージェント（私）は、このプロジェクトでの経験を通じて、以下の行動原則と学びを得ました。これは、将来のタスク実行において常に遵守し、改善に努めます。

1.  **過度な自動化・統一の回避**:
    *   ツールやスクリプトによる自動化や統一は、その効果とリスクを慎重に評価すべきです。特に、Markdownコンテンツの構造解析や自然言語の意図解釈など、AIの判断が介入する部分で完璧な自動化を目指すことは、複雑性を増し、柔軟性を損なう可能性があります。
    *   コードブロックや例示など、原文のままの方が理解しやすい、あるいは改変がリスクとなる箇所については、安易に統一せず、原文を尊重します。

2.  **テストフェーズにおける行動原則の明確化**:
    *   テストフェーズ（特にスキルの動作確認中）において、テスト対象のスキルファイルそのものを変更する行動は、テストの目的に反し、混乱を招きます。テスト中は、テスト対象の挙動の評価を最優先とし、スキルファイル自体の修正はテスト完了後の実装フェーズで行うべきです。

3.  **ユーザーへの意図確認の徹底**:
    *   方針の解釈や適用範囲について曖昧な点がある場合、安易な仮定に基づいて行動するのではなく、まずユーザーにその意図を明確に確認します。これにより、誤解に基づく無駄な作業や、ユーザーの意図しない変更を回避します。

4.  **Gitワークツリー操作における `dir_path` の徹底 (2026年2月18日)**:
    *   `run_shell_command` や `write_file` などのツールは、実行ごとにプロセスのコンテキスト（カレントディレクトリ）がリセットされる性質を持つ。
    *   `using-git-worktrees` スキルで作成した隔離環境（ワークツリー）で作業する場合、ツール呼び出し時に必ず `dir_path` （またはファイルパスのプレフィックス）を明示的に指定しなければならない。これを怠ると、意図せずメインのワークスペース（例: `main` ブランチ）に対して変更やコミットを行ってしまうリスクがある。

### 5. 記憶ファイル (`GEMINI.md`) の運用方針

*   **継続的な更新**: このファイルは、プロジェクトの進行に伴う新たな学び、最適なプラクティス、重要な決定事項、およびエージェントの行動に関する指針を継続的に追記・更新します。
*   **構造化**: 各項目は明確な見出しを設け、日付を付与することで、参照しやすく、履歴を追えるようにします。
*   **意思決定の記録**: 特定の選択肢やアプローチを採用した理由、トレードオフに関する情報なども記録し、将来のコード理解や意思決定に役立てます。
*   **透明性**: エージェントの行動原理や思考プロセスを外部から理解できる形で記述します。

### 6. 開発ワークフローの原則 (2026年2月15日)

#### 6.1. ブランチ・ワークツリー戦略 (作業分離の原則)

一貫性と安全性を確保するため、すべての開発・テスト作業は以下の原則に従って分離します。

1.  **フィーチャー開発**:
    *   新しい機能開発やスキル移植などの永続的な作業は、必ず専用のフィーチャーブランチ（例: `feat/add-new-feature`, `feat/port-some-skill`）を作成して行います。
    *   **ワークツリーの選択**: 
        *   通常の開発タスクでは、`using-git-worktrees`スキルを利用して専用のワークツリーを作成することを推奨します。これにより、複数の作業を並行して安全に進めることができます。
        *   **スキル開発・移植の場合**: 実装フェーズでは通常のブランチ（メインワークスペース）での作業を推奨します。これは、スキル認識のためのCLI再起動やパス指定（`dir_path`）の煩雑さを避け、実装プロセスをシンプルに保つためです。
    *   **重要:** ワークツリー内での作業中、`run_shell_command` 等のツールを使用する際は、常に `dir_path` にそのワークツリーのパスを指定してください。そうしない場合、コマンドはプロジェクトのルートディレクトリ（元のワークスペース）で実行されます。

2.  **テスト実行**:
    *   実装された機能やスキルのテストは、開発用のブランチやワークツリーからさらに分離された、**一時的なテスト環境**で行います。
    *   テスト環境の作成には、`using-git-worktrees`スキルを再度利用し、テスト専用のブランチ（例: `test/run-skill-test`）を持つ新しいワークツリーを作成する方法を推奨します。
    *   テスト完了後、テストに使用したワークツリーとブランチは、`git worktree remove`や`git branch -D`コマンドで完全にクリーンアップします。これにより、テストによる一時的な成果物がリポジトリの履歴を汚すことを防ぎます。

3.  **テストフレームワークの選択**:
    *   このプロジェクトでは、生成AIスキルの開発・テストが主目的であり、Python標準の `unittest` フレームワークを優先して使用する。外部ライブラリへの依存を最小限に抑え、標準的な環境での動作を保証するためである。

#### 6.2. セッション管理の原則 (2026年2月28日更新)

コンテキストの管理とスキルの認識を適切に行うため、以下の2つの操作を明確に区別して運用します。

1.  **セッションのリフレッシュ (Session Refresh)**:
    *   **目的**: 肥大化した会話履歴を破棄し、コンテキスト（トークン使用量）を節約してパフォーマンスを向上させる。
    *   **方法**: 現在の会話を完全に終了し、新しいセッションを開始する（`--resume` を使用しない）。
    *   **必須事項**: 次のセッションでスムーズに作業を再開できるよう、現状の要約と次のタスク提案を含む**再開用プロンプト**を生成して提示する。
    *   **タイミング**: 大規模なタスク（例：一つのスキル移植）の完了時や、会話が長くなり応答が遅延し始めた時。

2.  **スキルの認識・再起動 (Skill Restart)**:
    *   **目的**: 新しく作成・変更したスキルファイル (`SKILL.md`) を Gemini CLI に読み込ませ、認識させる。
    *   **方法**: `gemini --resume latest` を実行して、**履歴を保持したまま**プロセスを立ち上げ直す。
    *   **必須事項**: 通常、新しいワークツリーディレクトリ等へ移動した直後に実行する。履歴が維持されるため、再開用プロンプトは不要。
    *   **タイミング**: スキルのテストフェーズへの移行時、または隔離環境（ワークツリー）での作業開始時。

#### 6.3. 作業完了時のシーケンス (2026年2月28日更新)

開発作業が完了し、ブランチをマージ・クリーンアップする際は、以下の順序を厳守することで知見の永続化とシステムの自己改善を確実にします。

1.  **振り返り (Reflection)**: 作業内容を振り返り、得られた知見や反省点を整理する。
2.  **プロセス改善の反映 (Process Improvement)**: 振り返りで得られた知見を、関連するスキル (`SKILL.md`) や `GEMINI.md` に反映（追記）する。必要に応じて `writing-skills` を使用する。
3.  **セッション管理 (Session Management)**: すべての記録が完了した後に、必要に応じてセッションのリフレッシュまたは再起動を行う。

### 7. 自己改善システムの導入 (2026年2月27日)

AIエージェントが、実戦で得られた知見を自律的かつ安全に指示書（SKILL.md）へ反映し、自身のパフォーマンスを継続的に向上させるためのシステムを導入しました。

**主要な機能:**
1.  **スキルリセットスクリプト (`scripts/reset_skill.py`):**
    *   `SKILL.md`内の`## ローカル・アダプテーション (Gemini固有)`セクションとその内容を削除し、スキルを純粋な移植版の状態にリセットします。
2.  **開発完了時の改善提案 (`finishing-a-development-branch`):**
    *   開発ブランチの完了時に、AIエージェント自身が作業内容を振り返り、スキル改善の機会を提案するようになりました。必要に応じて`writing-skills`を起動し、スキルを更新できます。
3.  **スキルアップデート時のローカル変更保護 (`port-superpowers-skill`):**
    *   移植済みスキルを元のリポジトリからアップデートする際に、既存のローカル・アダプテーションをバックアップし、新バージョン適用後に安全に再結合する手順がSOPに追加されました。

**目的:**
*   AIエージェントの継続的な自己学習とパフォーマンス向上。
*   ローカルでのスキル改善と、移植元リポジリからのアップデートとの安全な両立。
*   緊急時のスキル状態の迅速なリセット機能の提供。

### 8. 実戦からの学び (2026年3月8日更新)

1. **`dir_path` 指定の徹底**:
    * 隔離環境（ワークツリー）で作業する際は、ファイルI/Oを伴うすべてのツールコールにおいて、必ず `dir_path` を明示しなければならない。
2. **スキルへの手順明記の重要性**:
    * プロセスを定義するだけでなく、具体的な手順やフォーマットを明記することが不可欠である。
3. **ワークツリー内でのパス認識とプロジェクトルート**:
    * サブセッションでは相対パスの解決が異なる可能性があるため、起動プロンプトや指示書には「絶対パス」を原則とする。
4. **サブセッションにおける役割分担の原則**:
    * **責務の分離**: サブセッションは「タスクの完了とコミット」までを責務とし、統合は親セッションが行う。
    * **ハイブリッド連携プロトコル**: スクリプト（Python）は YAML Frontmatter 等の「構造・枠組み」の生成に徹し、AI はその中の「文脈・意味（任務や成果）」を埋める役割を担う。この境界線を実装計画段階で物理的（入出力関係）に定義することが、実装品質を左右する。
5. **ワークスペース制限とシェル操作の活用**:
    * Gemini CLI のセキュリティ制限（`read_file`/`write_file` の対象外パス）を回避するため、グローバル領域へのアクセスには `run_shell_command` (cat, cp) を使用し、「一時ファイル経由の上書き」パターンを標準とする。
6. **隔離環境（ワークツリー）とツールの同期**:
    * ワークツリー作成後にメインブランチでツール（スクリプト等）を修正した場合、その変更は既存のワークツリーには反映されない。テストサイクルごとにワークツリーを新しく作り直すことが、最も確実な検証手段である。
7. **高品質プロセスの連鎖**:
    * 複雑な設計の修正には `brainstorming` (再設計) -> `writing-plans` (詳細化) -> `subagent-driven-development` (2段階レビュー) -> `verification-before-completion` (証拠提示) というフルコースのスキル適用が極めて有効である。

### 9. テストおよびスクリプトの技術選定方針 (2026年3月2日)


本プロジェクトでは、環境構築のオーバーヘッドを最小限に抑え、ポータビリティを最大限に高めるため、以下の技術選定方針を厳守する。

1. **標準ライブラリの優先使用**:
    *   外部パッケージ（`pip` や `npm` によるインストールが必要なもの）への依存を可能な限り排除する。
    *   Python では標準の `unittest` を使用し、Node.js への移行後も標準の `node:test` を使用する。
2. **「低オーバーヘッド」が最大の機能**:
    *   複雑なテストフレームワークを導入するよりも、どの環境でも即座にテスト・実行できることを優先する。これは、新しい環境でスキルを素早く展開するための戦略的選択である。
3. **将来の Node.js 移行**:
    *   Gemini CLI 環境との親和性を考慮し、長期的には Node.js への一本化を目指す。ただし、その際も「標準機能のみで完結させる」という原則は維持する。
