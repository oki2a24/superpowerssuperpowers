## 6. 将来の展望 (Future Roadmap)

本プロジェクトは継続的に進化します。現在、以下の改善を検討しています。

- **Node.js への移行検討**: Gemini CLI 自体が Node.js ベースであることを活かし、運用スクリプトやテストを Node.js (`node:test` ランナーおよび `node:assert` モジュール) へ順次移行する。
  - **目的**: 環境の統一によるセットアップ負荷の軽減と、Gemini CLI との親和性の向上。
  - **方針**: 外部ライブラリへの依存を避け、Node.js 標準機能のみで完結させる。
- **サブセッション連携の自動化**: `using-git-worktrees` で起動したサブセッション（ワークツリー内の別セッション）との連携を、よりシームレスかつ自律的に行うための仕組みづくり。
- **「異論・エラー」からの自己改善ループの強化**: ユーザーからの「ストップ」「疑問」「エラー報告」を、単なる修正対象ではなく「スキルの不備」として捉え、自律的に `SKILL.md` や `GEMINI.md` を改善するトリガーの精緻化。

```

### 9. テストおよびスクリプトの技術選定方針 (2026年3月2日)

本プロジェクトでは、環境構築のオーバーヘッドを最小限に抑え、ポータビリティを最大限に高めるため、以下の技術選定方針を厳守する。

1. **標準ライブラリの優先使用**:
    *   外部パッケージ（`pip` や `npm` によるインストールが必要なもの）への依存を可能な限り排除する。
    *   Python では標準の `unittest` を使用し、Node.js への移行後も標準の `node:test` を使用する。
2. **「低オーバーヘッド」が最大の機能**:
    *   複雑なテストフレームワークを導入するよりも、どの環境でも即座にテスト・実行できることを優先する。これは、新しい環境でスキルを素早く展開するための戦略的選択である。
3. **将来の Node.js 移行**:
    *   Gemini CLI 環境との親和性を考慮し、長期的には Node.js への一本化を目指す。ただし、その際も「標準機能のみで完結させる」という原則は維持する。
```
