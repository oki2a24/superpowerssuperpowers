# 次世代 GPAC (v4) & サブエージェント連携の設計思想 (Insights from Node.js Migration Session)

## 1. 背景と動機 (The "TODO" Disaster)
2026-03-12 の Node.js 移行セッションにおいて、`scripts/todo.mjs` の単体テストを `generalist`（サブエージェント）に委譲して実行した際、テストコードがハードコードされたパス `.gemini/tasks/` 下のファイルを削除。これにより、本セッションの進捗管理を行っていた実データが消失する事故が発生した。

この「足元の破壊」から、**「共有環境での迅速な委譲（Subagent）」** と **「隔離環境での安全な委譲（gemini-sub）」** の戦略的使い分けと、その境界を溶かすための連携向上の必要性が浮き彫りになった。

## 2. コア・コンセプト (The Four Pillars of Connectivity)

### A. 統合インターフェース (Unified Delegate Interface)
- **概念**: `generalist` ツールに `--isolated`（または `--worktree`）フラグを導入。
- **物理的挙動**: ユーザーは一つの「委譲」アクションに対し、安全性を優先するか（隔離セッション）、速度を優先するか（現在のセッション）をオプションで切り替える。複雑なプロトコル（spawn / import）の裏側をエージェントが自律的に制御する。

### B. コンテキスト・ハイドレーション (Strategic Context Hydration)
- **概念**: 隔離セッション（`gemini-sub`）起動時における、メインセッションの「最新の教訓」の自動同期。
- **物理的挙動**: `spawn` 時、メインセッションの `GEMINI.md`（SSOT）や直近の「振り返り」をスナップショット化し、サブセッションの初期記憶に流し込む。「隔離されているが、同じチーム（知識共有）」という状態を実現する。

### C. 自律的 SSOT フィードバック・ループ (Autonomous SSOT Update)
- **概念**: サブセッションの「痛み（失敗）」を、メインセッションの「資産（規律）」へ自動還元する。
- **物理的挙動**: `import` 時、報告書の「Engineering Insights」セクションを自動解析し、プロジェクトの `GEMINI.md` への追記（物理的ガードレールの追加）をユーザーに提案する。

### D. 環境変数による物理的 I/O 境界の強制 (Sandboxed I/O Discipline)
- **概念**: ツール側の密結合を排除し、テスト時の安全性をコードレベルで保証する。
- **物理的挙動**: 物理パスを環境変数（例：`GEMINI_TASK_DIR`）で上書き可能にすることを、プロジェクトの開発規律（`GEMINI.md`）に明記。テスト時には一時ディレクトリの使用を強制し、実データの汚染を 100% 回避する。

## 3. 教訓の物語：なぜ `gemini-sub` は必要なのか
サブエージェントは「私の手足」として高速に動作するが、私の「部屋」に居るため、家具（進捗管理ファイルなど）を壊す可能性がある。一方、`gemini-sub` は「別の作業小屋」を用意する。重たい作業、あるいは今回のようにファイルを激しく操作するテストにおいては、小屋を分けて作業することの安全価値は、コンテキスト連携のコストを上回る。
