# デザインドキュメント: todo.py の Node.js への移行

## 1. 目的
Gemini CLI のタスク管理ユーティリティ `scripts/todo.py` を Node.js (v25.8.0, ESM) に移植します。これにより、プロジェクト内の Node.js への統一を進め、実行環境の依存関係を最小化します。

## 2. アーキテクチャとデータフロー

### 2.1 コンポーネント
- **`scripts/todo.mjs`**: メインロジック。
    - `TASK_DIR`: `.gemini/tasks`
    - `getBranchName()`: `git branch --show-current` または `git rev-parse --abbrev-ref HEAD` を使用。
    - `getTodoPath()`: ブランチ名からファイルパスを生成。
    - `init(title)`: ファイルの初期化とヘッダー書き込み。
    - `add(task)`: タスクの追記。
    - `start(pattern)`: 正規表現によるステータス更新 (`[ ]` -> `[/]`)。
    - `done()`: ステータス更新 (`[/]` -> `[x]`)。
    - `show()`: ファイル内容の表示。
    - `main()`: コマンドライン引数の処理と終了コード制御。
- **`tests/test_todo.mjs`**: `node:test` を使用した単体テスト。

### 2.2 データフロー
1.  **入力**: `process.argv` からサブコマンドと引数を取得。
2.  **ブランチ取得**: `spawnSync('git', [...])` で現在のブランチ名を取得し、ファイル名に安全な形式に変換。
3.  **ファイル操作**: `fs.readFileSync`, `fs.writeFileSync`, `fs.appendFileSync` を使用。
4.  **状態遷移**: 文字列置換により `[ ]` (未着手) -> `[/]` (進行中) -> `[x]` (完了) を管理。

## 3. エラーハンドリングと制約
- **Git不在/非リポジトリ**: ブランチ名取得に失敗した場合は `"default"` を使用。
- **重複開始の禁止**: 既に `[/]` が存在する状態で `start` を実行した場合は、エラーメッセージを表示して終了コード 1。
- **パターン不一致**: `start` で一致するタスクがない場合は終了コード 1。
- **標準ライブラリ**: `npm install` 禁止。

## 4. テスト戦略 (TDD)
1.  **RED**: `tests/test_todo.mjs` で、各サブコマンドの振る舞いを定義。
    - `init` がヘッダーを生成すること。
    - `add` がタスクを追記すること。
    - `start` が正しく状態を遷移させ、二重起動を防ぐこと。
    - `done` が進行中タスクを完了させること。
    - `show` が正しいフォーマットで出力すること。
2.  **GREEN**: `scripts/todo.mjs` を実装し、テストをパスさせる。
3.  **REFACTOR**: 日本語コメントの充実、コードの共通化。

## 5. 日本語コメントの継承
Python 版の各関数の Docstring およびテストの目的説明を JSDoc 形式で漏らさず移植します。特に `start` における二重起動禁止の意図などを明確にします。
