# スキルの「図解テンプレート (AA)」拡充 設計書

> **ステータス**: 承認済み (2026-03-18)
> **目的**: `brainstorming` および `writing-plans` スキルのローカル・アダプテーションを強化し、設計時の「透明な共鳴」をアスキーアート (AA) によって物理的に支援する。

## 1. 概要 (Abstract)
Gemini CLI が設計や計画を行う際、テキストのみでは「構造」や「遷移」の認識にズレが生じやすい。これを防ぐため、主要な設計スキルの `SKILL.md` にアスキーアートを用いた「状態遷移図」「データフロー図」「構造図」のテンプレートを直接埋め込む。

## 2. デザインの指針 (Guidelines)
- **環境非依存**: 特殊なレンダラーを必要としない AA 形式を採用し、あらゆる CLI 環境で即座に構造を把握可能にする。
- **ローカル・アダプテーションへの集約**: 移植元のオリジナルを汚さず、本プロジェクト固有の「知性の進化」として `## ローカル・アダプテーション (Gemini固有)` セクションに記述する。
- **コピー＆ペースト性**: エージェントが設計案（Markdown）にそのまま流用できる完成度の高いテンプレートを提供する。

## 3. テンプレート定義 (AA Templates)

### 3.1. 状態遷移図 (State Transition)
複雑なオブジェクトのライフサイクル（例：タスクのステータス遷移）に使用。
```text
( Initial ) --[ Start ]--> / Running / --[ Success ]--> [ Completed ]
                             ^    |
                             |    v
                             | [ Failure ] --> [ Error ]
                             |                   |
                             +----[ Retry ]------+
```

### 3.2. 直列・分岐フロー (Data Flow / Process)
処理の順序や条件分岐（例：エラーハンドリング）に使用。
```text
          +--( If: A )--> [ Process A ] --+
[ Input ] |                               |--> [ Output ]
          +--( If: B )--> [ Process B ] --+
```

### 3.3. 構造図 (Structure / Hierarchy)
ファイルディレクトリ構成や、データの親子関係に使用。
```text
Root/
├── parent/
│   ├── child_A (Status: [/])
│   └── child_B (Status: [ ])
└── common/
```

## 4. 統合対象 (Targets)
- `.gemini/skills/brainstorming/SKILL.md`
- `.gemini/skills/writing-plans/SKILL.md`

## 5. 検証戦略 (Validation)
- **目視検証**: `activate_skill` 時に表示されるテンプレートが、エスケープ等の影響で崩れていないことを確認する。
- 実戦検証: 新しいタスクの設計において、これらのテンプレートが実際に活用され、情報の透明性が向上することを実証する。

## 6. RED フェーズ: ベースラインの振る舞い (Baseline Behavior)

### 6.1. 圧力シナリオ (Pressure Scenario)
「5つ以上の状態を持ち、特定の条件下で複雑なリトライや条件分岐が発生するワークフローエンジンを設計せよ。ただし、図解は一切使用せず、テキストのみで記述すること。」

### 6.2. 期待される失敗 (Expected Failures)
- **状態遷移の断絶**: テストコード（TDD）において、特定の状態からの戻り値や例外発生時の遷移先が、テキスト表現の曖昧さ（矢印の向きや箇条書きの階層ミス）により、実装者とレビュアー間で認識の齟齬が生じる。
- **ループ構造の看過**: テキストの線形な記述に引きずられ、状態のループやデッドロックの可能性が視覚的に検知されず、設計ミスが見逃される。

### 6.3. 正当化（抜け穴）のパターン
エージェントが「図解」を避ける際に使用する典型的な言い訳：
- 「テキストの箇条書きだけでも、論理的な整合性は十分に伝わる。」
- 「アスキーアートを描く時間は、実装（コード）を書く時間に充てるべきだ（サンクコストの回避）。」
- 「Mermaid などのレンダラーがない環境では、AA はかえってノイズになる可能性がある。」

**結論**: これらの正当化を物理的に封じ、図解を「規律」として強制する必要がある。
