# デザインドキュメント：完了宣言プロトコルと物理バリデータの導入

## 1. 背景と目的
AI エージェントが個別のタスク（関心事）を完了させた際、プロジェクト全体との整合性（後始末、ドキュメント同期、バージョニング）の確認が属人化（AI の主観頼み）しており、抜け漏れが発生するリスクがある。本設計では、憲法による規律（DoD）と、スクリプトによる物理的な差分分析を組み合わせることで、高品質なアウトプットを体系的に保証する。

## 2. 物理バリデータ `scripts/finish_check.mjs`

### 機能要件
- **比較基準の柔軟性**: 引数として比較対象のコミット/ブランチを指定可能にする（デフォルト: `origin/main`）。
- **未完了作業の検知**: ステージング漏れや一時ファイルの消し忘れを `git status` で確認。
- **作業全容の可視化**: 指定された基準から現在までの変更ファイルリストを表示し、AI に README 等との同期を促すヒントを与える。
- **矛盾の検知**: スキルや知見に変更があるにもかかわらず、バージョンファイル（`gemini-extension.json`）が更新されていない「不自然な状態」を警告する。

### 入出力仕様
- **Input**: `[base_ref]` (Optional, string)
- **Output**: 
    - `[PASS/FAIL]` Git のクリーン状態。
    - `[DELTA]` 変更されたファイルのサマリ。
    - `[ALERT]` バージョン更新漏れ等の警告。

## 3. 憲法 (`GEMINI.md`) への統合

### 完了の定義 (Definition of Done / DoD) の義務化
AI は完了を宣言する直前に、以下の「完了プロトコル」を自律的に踏まなければならない。

1.  **物理チェックの実行**: `node scripts/finish_check.mjs` を実行し、客観的な事実を確認する。
2.  **チェックリストの報告**: 以下の 4 項目について、実行結果に基づきユーザーに報告する。
    - **Cleanup**: 一時ファイル、一時ブランチ等の削除状況。
    - **Doc Sync**: 物理ファイル構成と `README.md` 等の同期状況。
    - **Versioning**: 変更内容に応じた `gemini-extension.json` の更新状況。
    - **Git Integrity**: 未コミット変更の有無。

## 4. 期待される効果
- AI が「たまたま思い出す」のではなく、「事実を突きつけられて対応する」状態が作れる。
- セッションを跨いだ作業であっても、物理的な差分に基づいた正確な整合性チェックが可能になる。
- ユーザー（人間）が AI の「完了の質」を信頼できるようになる。
