# 2026-03-31 Session Handoff Core Skill Design

## 1. 🎯 目的 (Objective)
セッションの肥大化やコンテキストの切り替えが発生した際、現在の思考状態、物理的な作業状況、および「意思決定の背景 (Why)」を構造化し、次のセッションへ無損失で引き継ぐためのコアスキル `session-handoff` を定義する。

## 2. 🧩 解決する課題 (Problem Statement)
- **情報の過剰な圧縮**: 従来の要約では、試行錯誤の過程や「なぜその設計にしたか」という文脈が欠落し、再開したAIが誤解を生むことがあった。
- **物理的アンカーの不在**: 作業中のファイル、行番号、直近の実行コマンドなどの「手触り」が失われ、再開時の立ち上がりが遅れる。
- **AIの「楽をしたい」性質**: 生成AIは本能的に情報を要約・省略しようとするため、重要な詳細が削ぎ落とされるリスクがある。

## 3. 🛡️ 解決策：Documentation TDD によるガードレール
スキルの作成において `writing-skills` の「Documentation TDD」を適用し、以下のガードレールを敷く。

### 戦略 (Strategy)
1. **RED (失敗の定義)**: AIが意図的に情報を省略・要約し、再開に支障が出る「失敗シナリオ」を定義する。
2. **GREEN (最小限の指示)**: 失敗を阻止するための最小限のテンプレートと規律（Whyの明文化、物理的アンカーの記録）を `SKILL.md` に記述する。
3. **REFACTOR (防弾化)**: AIが「これは自明だから省略してもいい」と言い訳（正当化）するパターンを特定し、それを禁止事項として明示的に塞ぐ。

## 4. 📝 ハンドオーバー・テンプレート (Proposal)
- **最終目標 (Ultimate Goal)**: 1行で。
- **完了事項と「意思決定の背景」 (Done & Why)**: 何をしたか＋なぜそうしたか（試行失敗も含む）。
- **物理的状態 (Physical Anchor)**: ファイル、行番号、直近のコマンド（成功/失敗）。
- **次の具体的なアクション (Next Steps)**: 迷わず次に叩くべきコマンド。
- **再開用プロンプト (Resumption Prompt)**: ユーザーがコピペして即座に再開できる文言。

## 5. 🛠️ 運用規律 (Core Discipline)
- **意図への反応**: ユーザーが「切り替えたい」「重くなってきた」「別の話題へ」といった意図を示した際に即座に起動する。
- **圧縮の禁止**: 「自明なことは省略する」というAIの性質を、スキルの規律によって物理的に禁止する。

## 6. 📅 ロードマップ (Execution)
- [ ] RED: 失敗シナリオでの実行と正当化の文書化。
- [ ] GREEN: 最小限の `SKILL.md` 作成と合格確認。
- [ ] REFACTOR: 言い訳（抜け穴）の封鎖。
