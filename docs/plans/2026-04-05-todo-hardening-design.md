# todo.mjs 堅牢化と操作性の改善 設計書 (2026-04-05)

## 概要
`scripts/todo.mjs` における操作ミスの削減、状態遷移の柔軟化、および AI エージェントによる確実なタスク管理を実現するための設計。

## 課題 (Problem Statement)
- **タスク指定の不確実性**: 正規表現（pattern）による指定は曖昧で、重複や誤認を招きやすい。
- **状態遷移の硬直性**: 既に実行中のタスクがある場合のエラーが作業フローを妨げる。
- **AI への不親切**: エラーメッセージが不親切で、AI が次に何をすべきか（show で ID を確認するなど）を判断しにくい。

## 解決策 (Proposed Solution)

### 1. ID ベースの一意な操作 (ID-Based Operation)
- **動的 ID 割り当て**: `show` コマンド実行時に、上から順に連番（1, 2, 3...）を付与。
- **表示形式**: `- [ ] (1) Task Name` のように、Markdown リスト内で ID を明示。
- **一意の指定**: `start 1`, `done 2` のように ID で操作可能にする。

### 2. 状態遷移の柔軟化 (Auto-suspend)
- **挙動**: 実行中（`[/]`）のタスクがある状態で別のタスクを `start` した場合、自動的に前のタスクを未着手（`[ ]`）に戻し、新しいタスクを開始する。
- **例外**: 子タスクを開始する場合は、親タスクは `[/]` のまま維持する（階層的なフォーカス）。

### 3. AI フレンドリーなガイドとインターフェース
- **`scripts/TODO_GUIDE.md`**: AI が操作前に一読すべき「最短リファレンス」を作成。
- **構造化 `--help`**: 単なるテキストではなく、AI がパースしやすい、あるいは「規律」として認識しやすい形式に刷新。
- **ヒント付きエラーメッセージ**: エラー発生時に「ID を確認するために show を実行してください」等の具体的なアクションを提示。

## 実装規律 (Implementation Discipline)
- **TDD (Test-Driven Development)**: すべての機能変更は、まず失敗するテストを作成することから開始する。
- **回帰テストの徹底**: 各タスクの完了時に、追加したテストだけでなく `todo.mjs` に関連するすべての既存テストがパスすることを確認する。
- **最終検証**: 全タスク完了後、プロジェクト全体の自動テストを実行し、デプロイ可能な状態であることを物理的に証明する。

## 成功基準 (Success Criteria)
- `todo.mjs show` で ID が正しく表示されること。
- `todo.mjs start <ID>` でタスクが開始できること。
- 実行中タスクがある状態で別のタスクを `start` してもエラーにならず、前のタスクが `[ ]` に戻ること。
- `--help` が AI にとって理解しやすい構造（セマンティックな見出し等）になっていること。
- `scripts/TODO_GUIDE.md` が作成され、AI がその存在を認識できること。
