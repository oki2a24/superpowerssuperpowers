# デザイン仕様：フラクタルな原子性に基づくスキルアップデート・プロトコル

## 1. 目的
スキル移植やアップデート（特に `port-superpowers-skill`）において、1つのファイル内にある複数の独立した変更（関心事）を一度に適用してしまう問題を解決する。変更を最小単位（Atomic Concern）に分解し、一つずつ物理的な差分（`git diff`）を用いて人間によるレビューを受ける「フラクタルな原子性」を確立する。

## 2. 変更内容

### 2.1. SOP の更新 (`.gemini/skills/port-superpowers-skill/SKILL.md`)
「高度なスキルアップデート・プロトコル (Upstream Sync)」のステップ3を以下の通り再定義する。

#### ステップ3: フラクタルな原子性に基づく実装 (Atomic Execution)
分解した各関心事（Atomic Concern）に対し、以下のループを**厳格に**繰り返す：
1.  **単一関心事の選択**: `todo.mjs` に登録した関心事から、次に着手する1つを明確に宣言して選択する。
2.  **ターゲットを絞った更新**: `replace` ツールを使用し、その関心事に関連する箇所**のみ**を修正する。論理的に独立した複数の修正を1つのツール呼び出しに同梱してはならない。
3.  **物理的差分の提示**: 修正直後に必ず `git diff` を実行し、変更箇所をユーザーに明示する。
4.  **人間によるレビューゲート**: ユーザーに差分を確認してもらい、承認（`ok` 等）を得るまで、次の関心事や次のファイル操作に進まない。
5.  **知見の同期**: 独自の適応（Observations）の更新も、独立した関心事として同様のループを適用する。

### 2.2. プロジェクト全体知見の追加 (`.gemini/observations/GEMINI.md`)
普遍的な規律として L4（Project Local）に以下の項目を追加する。

- **フラクタルな原子性の遵守**: ファイル編集において「原子性（Atomic Integrity）」をフラクタルに適用せよ。1つのファイル内であっても、複数の独立した変更を1つの `replace` 呼び出しに同梱することを厳禁とする。
- **物理的証拠による共鳴**: 変更の意図を言葉で説明するだけでなく、常に `git diff` という物理的な証拠を提示し、ユーザーとの「透明な共鳴」を維持した状態で承認を得なければならない。
- **効率より規律**: ターン数を節約するために変更をまとめることは、不純物の混入を招く「規律の欠如」とみなす。正確性とレビューの容易さを常に優先せよ。

## 3. ワークフローと制約

### 3.1. レビューとコミット
- **レビュー単位**: `replace` 1回（＝1つの関心事）ごと。
- **コミット単位**: 全ての関心事のレビューが完了した後、一連の変更をまとめてコミットする（作業中の差分を積み上げる形式）。

### 3.2. 成功基準
- 私が 1 つの `replace` で複数の論理的に独立した変更を行おうとした場合、ユーザーが即座に「規律違反」として指摘できる状態になっていること。
- 全ての変更が `git diff` によって人間が容易に読み取れる粒度で提示されること。

## 4. スペック自己レビュー

1.  **プレースホルダのスキャン**: 全て定義済み。
2.  **内部の一貫性**: 「効率より規律」という原則が、全ステップで一貫して適用されている。
3.  **スコープチェック**: `port-superpowers-skill` だけでなく、プロジェクト全体の知見（L4）に反映させることで、将来的な全作業に適用可能。
4.  **曖昧さのチェック**: 「関心事（Concern）」の定義がプロジェクトごとに揺れる可能性があるが、`todo.mjs` での分解プロセスを前提とすることで具体性を担保している。

## 5. 意思決定の背景 (Rationale)

実装過程におけるユーザーフィードバックに基づき、以下の重要な方針転換を行った：

- **git diff の視認性問題**: 当初の設計では変更後に `git diff` でレビューを受ける予定だったが、ターミナル上での色の欠如や操作コストにより、人間による査読が困難であることが判明した。
- **replace プレビューの採用**: Gemini CLI の `replace` ツールが提供する標準の色付きプレビュー機能が、最も視認性が高く、編集直前に確認できるため合理的であると判断した。
- **Pre-write Review への転換**: 「書き込んだ後に確認する」のではなく、「書き込む直前に色付き差分で承認を得る」というフローを、フラクタルな原子性の核心に据えることとした。
