# デザイン仕様：Upstream Sync 堅牢化プロトコル

## 1. 目的
`obra/superpowers` (Upstream) の更新を、Gemini CLI 向けにカスタマイズされた日本語スキル群へ、効率的かつ安全に同期する。実質的なロジックの改善や構造の変更を、`port-superpowers-skill` のプロセスを再利用して反映しつつ、セッションの健全性を維持する。

## 2. コア・プロトコル (Unified Logic & Structure Sync)

### 2.1. ターゲットの特定 (Targeting Scope)
- **SSOT (唯一の真実)**: `docs/superpowers_ports.md` に記載されているスキルおよびコンポーネントのみを同期対象とする。
- **除外ルール**: 本プロジェクトオリジナルのスキル（`.gemini/skills/` 内や、移植履歴にないもの）は、いかなる場合も同期対象外とし、変更を加えない。
- **包含範囲**: `SKILL.md` に加え、スキルが参照するスクリプト、テンプレート、エージェント定義等の関連ファイルも差分スキャンの対象に含める。

### 2.2. 差分インベントリ (Inventory)
移植元の前回同期コミットと最新コミットを比較し、以下の3カテゴリで変更を分類する：
- **Modified (更新)**: 既存スキルのロジック・記述・関連ファイルの変更。
- **Added (新規)**: 新しいスキル、補助スクリプト、プロンプトテンプレートの追加。
- **Deleted (削除)**: ファイルの廃止、移動、または統合による削除。

### 2.3. 適応と日本語化の規律 (Syncing SOP)
Modified および Added の全てのファイルに対し、`port-superpowers-skill` の SOP を再利用する：
1. **Gemini 適応分析 (Porting Step 3)**: 差分内の Claude 固有要素（Digraph, Task等）を Gemini CLI 形式（Mermaid, run_shell_command等）へ変換する。
2. **日本語ドラフト (Porting Step 4)**: 英語の変更意図を抽出（ガードレール、手順、禁止事項、フロー、思想）し、既存の日本語コンテキストへ適切にマージする。

### 2.4. 構造同期の規律 (Structural Sync)
- **削除**: 構成の整合性を優先し、上流で廃止されたファイルはローカルでも不要な場合に限り削除を実行する。

### 2.5. 同期提案と人間による承認 (Review Gate)
- 全ての変更内容（抽出されたロジック、構成変更の理由）を日本語で要約提示し、承認（OK）を得てから物理的なファイル操作を行う。

## 3. セッション管理規律 (Session Orchestration)

### 3.1. 難易度判定と分割の基準 (Splitting Criteria)
スキャン完了後、以下の「1セッション完結」条件に照らし合わせ、続行か分割かを判断する：
- **1セッション続行条件**:
    - 変更スキルの総数が 10 以下。
    - 実質的なロジック修正が 1 スキルあたり 3 箇所未満の軽微なものである。
    - 他のスキルや外部ツールへの破壊的な依存関係変更がない。
- **分割判断**: 上記を満たさない場合、またはコンテキストの肥大化が懸念される場合は、分割を提案する。

### 3.2. 引き継ぎプロトコル (Handoff Protocol)
セッションを分割する場合、`session-handoff` スキルを使用し、同期状況、適用済みの重要判断、次回の着手ポイントを要約した再開プロンプトを作成する。`todo.mjs` は使用しない。

## 4. 実装形態： `sync-upstream-skill`

### 4.1. スキルの役割
- 同期プロセスの管理と、規律の強制。
- `port-superpowers-skill` の SOP を「エンジンの心臓部」として呼び出し、同期プロセス全体をオーケストレートする。

### 4.2. 完了の定義 (DoD)
- 全スキルの同期が完了している。
- `docs/superpowers_ports.md` が最新のコミットハッシュで更新されている。
- 物理的な動作検証（スキルのロード確認）が完了している。

## 5. 次回以降のショートカット
- `activate_skill("sync-upstream-skill")` により、本プロトコルを即座に起動する。
