## 設計時の必須チェック項目

1.  **「往復チケット (Return Ticket)」の原則**:
    *   外部プロセスやセッションを起動・開始する（Open）機能を設計する際は、必ずその完了と「統合（Close）」のための具体的な手段（例: IMPORT コマンドや報告フォーマット）をセットで設計してください。

2.  **「引き返す勇気」の尊重**:
    *   実装段階（`writing-plans` 以降）であっても、ユーザー（または自分自身）が設計上の矛盾や違和感を感じた場合は、即座にこの `brainstorming` フェーズまで立ち戻ることを推奨してください。無理な実装の強行よりも、設計の再考こそが最短ルートです。

3.  **戦略的分割の検討 (Strategic Decomposition Check)**:
    *   大規模なアイデアを形にする際、単一のモノリシックな計画にするのではなく、**「独立して並行（または段階的）開発可能なサブプロジェクト」**への分割を積極的に提案してください。
    *   **分割基準**:
        *   **複雑性**: 1つの計画書で 5つ以上の主要コンポーネントを扱う場合。
        *   **依存関係**: バックエンド API とフロントエンド UI のように、境界が明確な場合。
    *   分割された各サブプロジェクトは、GPAC v3 (`session-coordination`) を介して個別のサブセッションで実行することを標準戦略として提示してください。

## AIエージェントへの指示 (Gemini固有)
- **[イメージ] 図解の規律 (Visual Discipline)**: 設計案の提示には、必ず以下の AA テンプレートを参考に構造（状態遷移、データフロー、ディレクトリ構成等）を可視化せよ。

### 1. 状態遷移図 (State Transition)
複雑なオブジェクトのライフサイクル（例：タスクのステータス遷移）に使用。
```text
( Initial ) --[ Start ]--> / Running / --[ Success ]--> [ Completed ]
                             ^    |
                             |    v
                             | [ Failure ] --> [ Error ]
                             |                   |
                             +----[ Retry ]------+
```

### 2. 直列・分岐フロー (Data Flow / Process)
処理の順序や条件分岐（例：エラーハンドリング）に使用。
```text
          +--( If: A )--> [ Process A ] --+
[ Input ] |                               |--> [ Output ]
          +--( If: B )--> [ Process B ] --+
```

### 3. 構造図 (Structure / Hierarchy)
ファイルディレクトリ構成や、データの親子関係に使用。
```text
Root/
├── parent/
│   ├── child_A (Status: [/])
│   └── child_B (Status: [ ])
└── common/
```
