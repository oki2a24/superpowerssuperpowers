# アイデアをデザインに落とし込む

## 概要

このスキルは、協調的な対話を通じてアイデアを完全に形成されたデザインと仕様に変換するのを支援します。

**AIエージェントへの指示**: まず、現在のプロジェクトのコンテキストを理解し、アイデアを洗練させるためにユーザーに質問を一つずつ投げかけます。アイデアを理解したら、デザインを小さなセクション（200-300語）で提示し、各セクションの後に問題がないか確認します。

## プロセス

### 1. アイデアの理解

*   **AIエージェントへの指示**: まず、現在のプロジェクトの状態（ファイル、ドキュメント、最近のコミットなど）を確認してください。
*   **AIエージェントへの指示**: ユーザーにアイデアを洗練させるための質問を一つずつ投げかけます。可能であれば多肢選択形式の質問を優先しますが、自由回答形式でも構いません。1つのメッセージにつき質問は1つだけにしてください。もし特定のトピックがより深い探求を必要とする場合は、それを複数の質問に分割します。
*   **AIエージェントへの指示**: アイデアの目的、制約、成功基準を理解することに焦点を当ててください。

### 2. アプローチの探求

*   **AIエージェントへの指示**: トレードオフを伴う2-3種類のアプローチを提案してください。
*   **AIエージェントへの指示**: あなたの推奨オプションとその理由を添えて、対話形式でオプションを提示してください。
*   **AIエージェントへの指示**: 推奨オプションから提示を開始し、その理由を説明してください。

### 3. デザインの提示

*   **AIエージェントへの指示**: 何を構築するか理解できたと確信したら、デザインを提示してください。
*   **AIエージェントへの指示**: デザインを200-300語程度の小さなセクションに分割してください。各セクションの後に、問題がないかユーザーに確認してください。
*   **AIエージェントへの指示**: アーキテクチャ、コンポーネント、データフロー、エラーハンドリング、テストについて記述してください。
*   **AIエージェントへの指示**: もし何か理解できない点があれば、立ち戻って明確化してください。

### 4. デザイン承認後

**ドキュメント化:**

*   **AIエージェントへの指示**: 検証されたデザインを`docs/plans/YYYY-MM-DD-<topic>-design.md`に書き込んでください。
*   <!-- TODO: elements-of-style:writing-clearly-and-concisely スキルの移植を検討 -->
*   **AIエージェントへの指示**: デザインドキュメントをGitにコミットしてください。

**実装（続行する場合）:**

*   **AIエージェントへの指示**: ユーザーに「実装の準備ができましたか？」と尋ねてください。
*   **AIエージェントへの指示**: 隔離されたワークスペースを作成するために、`activate_skill(name="using-git-worktrees")`を呼び出すことをユーザーに提案してください。
*   **AIエージェントへの指示**: 詳細な実装計画を作成するために、`activate_skill(name="writing-plans")`を呼び出すことをユーザーに提案してください。

## 重要な原則

*   **一度に一つの質問** - 複数の質問でユーザーを圧倒しないこと。
*   **多肢選択を優先** - 可能であれば、自由回答形式よりも回答しやすい。
*   **YAGNIを徹底** - すべての設計から不要な機能を削除する。
*   **代替案を探求** - 結論を出す前に常に2-3種類のアプローチを提案する。
*   **段階的な検証** - デザインをセクションに分けて提示し、それぞれを検証する。

### 設計時の必須チェック項目

1.  **「往復チケット (Return Ticket)」の原則**:
    *   外部プロセスやセッションを起動・開始する（Open）機能を設計する際は、必ずその完了と「統合（Close）」のための具体的な手段（例: IMPORT コマンドや報告フォーマット）をセットで設計してください。

2.  **「引き返す勇気」の尊重**:
    *   実装段階（`writing-plans` 以降）であっても、ユーザー（または自分自身）が設計上の矛盾や違和感を感じた場合は、即座にこの `brainstorming` フェーズまで立ち戻ることを推奨してください。無理な実装の強行よりも、設計の再考こそが最短ルートです。

3.  **戦略的分割の検討 (Strategic Decomposition Check)**:
    *   大規模なアイデアを形にする際、単一のモノリシックな計画にするのではなく、**「独立して並行（または段階的）開発可能なサブプロジェクト」**への分割を積極的に提案してください。
    *   **分割基準**:
        *   **複雑性**: 1つの計画書で 5つ以上の主要コンポーネントを扱う場合。
        *   **依存関係**: バックエンド API とフロントエンド UI のように、境界が明確な場合。
    *   分割された各サブプロジェクトは、GPAC v3 (`session-coordination`) を介して個別のサブセッションで実行することを標準戦略として提示してください。

### AIエージェントへの指示 (Gemini固有)
- **[イメージ] 図解の規律 (Visual Discipline)**: 設計案の提示には、必ず以下の AA テンプレートを参考に構造（状態遷移、データフロー、ディレクトリ構成等）を可視化せよ。

#### 1. 状態遷移図 (State Transition)
複雑なオブジェクトのライフサイクル（例：タスクのステータス遷移）に使用。
```text
( Initial ) --[ Start ]--> / Running / --[ Success ]--> [ Completed ]
                             ^    |
                             |    v
                             | [ Failure ] --> [ Error ]
                             |                   |
                             +----[ Retry ]------+
```

#### 2. 直列・分岐フロー (Data Flow / Process)
処理の順序や条件分岐（例：エラーハンドリング）に使用。
```text
          +--( If: A )--> [ Process A ] --+
[ Input ] |                               |--> [ Output ]
          +--( If: B )--> [ Process B ] --+
```

#### 3. 構造図 (Structure / Hierarchy)
ファイルディレクトリ構成や、データの親子関係に使用。
```text
Root/
├── parent/
│   ├── child_A (Status: [/])
│   └── child_B (Status: [ ])
└── common/
```
