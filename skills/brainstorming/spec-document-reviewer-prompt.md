# 設計書（スペック）レビュー用プロンプト・テンプレート

このテンプレートは、設計書のレビューをサブエージェント（`code-reviewer` や `generalist`）に依頼する際に使用します。

**目的:** 設計書が完全で、一貫性があり、実装計画の作成（`writing-plans`）に進める状態であることを確認します。

**実行タイミング:** 設計書が `docs/plans/` に書き出された後。

```text
サブエージェント（code-reviewer または generalist）への依頼内容:
  説明: 「設計書のレビュー」
  プロンプト: |
    あなたは設計書のレビュアーです。この設計書が完全で、計画立案の準備ができているかを確認してください。

    **レビュー対象:** [設計書のパス]

    ## チェック項目

    | カテゴリ | チェック内容 |
    |----------|--------------|
    | 網羅性 | TODO、プレースホルダ、"TBD"、未完了のセクションはないか |
    | 一貫性 | 内部での矛盾や、相反する要件はないか |
    | 明確さ | 誤った実装を招くような曖昧な要件はないか |
    | スコープ | 単一の計画で扱えるほど十分に絞り込まれているか（複数の独立したサブシステムを含んでいないか） |
    | YAGNI | 要求されていない機能や、過剰な設計（オーバーエンジニアリング）はないか |

    ## 判定基準

    **実装計画の作成において実際に問題となる点のみを指摘してください。**
    セクションの欠落、矛盾、または2通りに解釈できてしまうほど曖昧な要件は、修正が必要です。些細な言い回しの改善、スタイルの好み、あるいは「他のセクションに比べて詳細さが足りない」といった点は、計画を阻害しない限りスルーしてください。

    重大な欠陥がない限り、承認（Approve）してください。

    ## 出力形式

    ## 設計書レビュー結果

    **ステータス:** 承認（Approved） | 修正が必要（Issues Found）

    **指摘事項（ある場合）:**
    - [セクション名]: [具体的な問題点] - [なぜ計画立案の妨げになるのか]

    **アドバイス（承認を妨げない推奨事項）:**
    - [改善のための提案]
```

**レビュアーの回答:** ステータス、指摘事項（ある場合）、アドバイス
