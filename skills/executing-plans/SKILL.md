# 計画の実行

## 概要

書かれた実装計画をロードし、批判的にレビューし、タスクをバッチで実行し、バッチ間でレビューのために報告します。

**開始時の宣言:** 「この計画を実装するために `executing-plans` スキルを使用します。」

**注記:** Superpowers はサブエージェントを利用できる環境（Claude Code や Codex など）でより高いパフォーマンスを発揮します。Gemini CLI でもサブエージェントが利用可能な場合は、このスキルの代わりに `subagent-driven-development` を使用することを強く推奨します。

**コア原則:** アーキテクトレビューのためのチェックポイントを持つバッチ実行。

## プロセス (The Process)

### ステップ 1: 計画のロードとレビュー
1. **計画のロード**: ユーザーに「実行する計画ファイルへのパス」を尋ねて読み込みます。
2. **批判的レビュー**: 計画を読み込み、質問や懸念点（矛盾、曖昧さ、欠落）がないかを確認します。
3. **エスカレーション**: 懸念がある場合、作業を開始する前に必ずユーザーに質問してください。
4. **タスク化**: 懸念がない場合、**タスク管理ツール**を使用してタスクリストを作成します。
    *   **Native-First**: Gemini CLI のビルトイン機能（`write_todos` 等）を優先して使用し、セッション中の進捗を管理します。
    *   **Fallback**: 3つ以上の独立したタスクがある場合や、永続化が必要な場合は、`scripts/todo.mjs` を併用して同期してください。
    *   **IDRプロトコル**: `todo.mjs` を使用する際は、必ず `show --json` で ID を確認してください。

### 2. バッチ実行

*   **デフォルト**: 最初の3タスクをバッチとして実行します。
*   各タスクについて:
    1.  タスクを「実行中 (In Progress)」にマークします。
    2.  計画に指定された各ステップを正確に実行します（計画は一口サイズのステップで構成されています）。
    3.  指定された検証を実行します。
    4.  タスクを「完了 (Completed)」にマークします。

### 3. レポート

バッチが完了したら、以下を報告します。

*   実装された内容を示します。
*   検証の出力を示します。
*   「フィードバックの準備ができました。」とユーザーに伝えます。

### 4. 続行

フィードバックに基づき、以下を行います。

*   必要に応じて変更を適用します。
*   次のバッチを実行します。
*   完了するまで繰り返します。

### 5. 開発の完了

すべてのタスクが完了し、検証された後、以下を行います。

*   **必須のサブスキル**: 移植された `finishing-a-development-branch` スキルを `activate_skill` で起動してください。
*   そのスキルに従い、テストの検証、オプションの提示、選択したオプションの実行を行います。
*   **開始時の宣言**: 「実装を完了するために `finishing-a-development-branch` スキルを使用します。」と宣言してください。

## ヘルプが必要な場合の停止と質問

**以下の場合、直ちに実行を停止します。**

*   バッチの途中でブロッカーに遭遇した場合（依存関係の不足、テストの失敗、指示の不明瞭など）。
*   計画に重大なギャップがあり、開始できない場合。
*   指示が理解できない場合。
*   検証が繰り返し失敗する場合。

**推測するのではなく、明確化のために質問します。**

## 以前のステップへの再訪問

**以下の場合、レビュー（ステップ1）に戻ります。**

*   ユーザーがフィードバックに基づいて計画を更新した場合。
*   根本的なアプローチの再検討が必要な場合。

**ブロッカーを強行突破（force through）しないこと** - 停止して質問してください。

## 重要な注意点

*   まず計画を批判的にレビューします。
*   計画のステップに正確に従います。
*   検証をスキップしないでください。
*   計画が指示するスキルは `activate_skill` を使用して参照します。
*   バッチ間では、報告してユーザーのフィードバックを待機してください。
*   ブロッカーに遭遇したら直ちに停止し、推測で進めないでください。
*   **絶対厳守**: ユーザーの明示的な同意なしに、`main` または `master` ブランチで実装を直接開始してはなりません。

## 統合

**必要なワークフロースキル:**

*   **`using-git-worktrees`** - 隔離されたワークスペースを確保します（新規作成、または既存の検証）。
*   **`writing-plans`** - このスキルが実行する計画を作成します。
*   **`finishing-a-development-branch`** - すべてのタスク完了後に開発を完了します。
