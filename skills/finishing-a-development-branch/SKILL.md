# 開発ブランチの完了 (Finishing a Development Branch)

## 概要

実装作業の完了をガイドし、明確なオプションを提示して、選択されたワークフローを処理します。

**コア原則:** テストの検証 → オプションの提示 → 選択の実行 → クリーンアップ。

**開始時の宣言:** 「実装を完了するために `finishing-a-development-branch` スキルを使用します。」

## プロセス

### ステップ 1: テストの検証

オプションを提示する前に、**必ずテストがパスすることを確認**します。

```bash
# プロジェクトのテストスイートを実行
npm test / cargo test / pytest / go test ./...
```

**テストが失敗した場合:**
```text
テストが失敗しました（<N>件）。続行する前に修正が必要です：

[失敗の詳細を表示]

テストがパスするまで、マージ/PRに進むことはできません。
```

ここで停止します。ステップ 2 には進まないでください。

**テストがパスした場合:** ステップ 2 に進みます。


### ステップ 2: ベースブランチの決定

```bash
# 一般的なベースブランチを検索
git merge-base HEAD main 2>/dev/null || git merge-base HEAD master 2>/dev/null
```

またはユーザーに尋ねます: 「このブランチは `main` から分岐しました。これでよろしいですか？」

### ステップ 3: オプションの提示

実装が完了したことをユーザーに伝え、以下の 4 つのオプションを**正確に、かつ簡潔に**提示します。

```text
実装が完了しました。何をしますか？

1. ローカルで <base-branch> にマージする
2. プッシュしてプルリクエストを作成する
3. ブランチを現状維持する（後で自分で処理します）
4. この作業を破棄する

どのオプションを選びますか？
```

**説明は付け加えず**、オプションのみを簡潔に提示してください。

### ステップ 4: 選択の実行

#### オプション 1: ローカルでマージ

```bash
# ベースブランチに切り替え
git checkout <base-branch>

# 最新を取得
git pull

# フィーチャーブランチをマージ
git merge <feature-branch>

# マージ後の結果でテストを検証
<test command>

# テストがパスした場合、ブランチを削除
git branch -d <feature-branch>
```

その後、ワークツリーのクリーンアップ（ステップ 5）に進みます。

#### オプション 2: プッシュしてプルリクエストを作成

```bash
# ブランチをプッシュ
git push -u origin <feature-branch>

# プルリクエストを作成
gh pr create --title "<title>" --body "$(cat <<'EOF'
## 概要
<変更内容を 2-3 個の箇条書きで記述>

## テスト計画
- [ ] <検証手順>
EOF
)"
```

「ブランチ `<feature-branch>` はプッシュされました。PR のレビューとマージを待つ間、ワークツリー（存在する場合）は保持されます。」と報告し、次のステップ（振り返り）に進みます。

**ワークツリーのクリーンアップは行わないでください。**

#### オプション 3: 現状維持

「ブランチ `<name>` を現状維持します。ワークツリーも `<path>` に保持されます。」と報告します。

**ワークツリーのクリーンアップは行わないでください。**

#### オプション 4: 破棄

**最初に確認を得てください：**
```text
これは永続的に削除されます：
- ブランチ <name>
- すべてのコミット： <commit-list>
- <path> のワークツリー（存在する場合）

確認のために 'discard' と入力してください。
```

正確な確認を待ってください。

承認された場合：
```bash
git checkout <base-branch>
git branch -D <feature-branch>
```

その後、ワークツリーのクリーンアップ（ステップ 5）に進みます。

### ステップ 5: ワークツリーのクリーンアップ

**オプション 1, 4 の場合：**

**重要：** ワークツリーの削除コマンドは、必ずメインワークスペース（プロジェクトルート）に移動した状態で実行してください。`run_shell_command` の `dir_path` を指定せず、自分自身のディレクトリを削除しようとして発生するエラー（`Directory not empty` 等）を回避します。

ワークツリー（メインワークスペース以外）にいるか確認します：
```bash
git worktree list | grep $(git branch --show-current)
```

該当する場合、それを削除します：
```bash
git worktree remove <worktree-path>
```

**ローカルの補足 (Gemini固有):** `.gemini/tasks/TODO-<feature-branch>.md` が存在する場合、削除します。

**オプション 2, 3 の場合：** ワークツリーを保持します。レビュー対応やマージ待ち、後日の作業継続のためです。


### 振り返り提案とスキル改善の機会

開発作業が完了し、この経験から得られた知見を AI エージェント自身に学習させる機会があります。
- この開発セッションを通じて、何か新しい学びや改善点がありましたか？
- 例えば、特定のツールの使い方、ワークフローの最適化、新しい問題解決のアプローチなどです。

もしスキル（特にこの `finishing-a-development-branch` スキル自身や関連スキル）に改善を提案したい場合は、`writing-skills` スキルを起動して具体的な変更を検討したり、`session-retrospective` スキルを起動して知見を蒸留したりすることができます。

このステップで振り返りを行いますか？

## 参考情報

### クイックリファレンス

| オプション | マージ | プッシュ | ワークツリー保持 | ブランチ削除 |
| :--- | :---: | :---: | :---: | :---: |
| 1. ローカルマージ | ✓ | - | - | ✓ |
| 2. PR 作成 | - | ✓ | ✓ | - |
| 3. 現状維持 | - | - | ✓ | - |
| 4. 破棄 | - | - | - | ✓ (強制) |

### よくある間違い

- **テスト検証のスキップ**
  - **問題**: 壊れたコードをマージしたり、失敗する PR を作成したりする。
  - **修正**: オプションを提示する前に、常にテストがパスすることを確認する。
- **オープンエンドな質問**
  - **問題**: 「次は何をすべきですか？」といった曖昧な質問。
  - **修正**: 正確に構造化された 4 つのオプションのみを提示する。
- **ワークツリーの自動クリーンアップ**
  - **問題**: 後で必要になる可能性があるワークツリーを削除してしまう（オプション 2, 3）。
  - **修正**: オプション 1 と 4 の場合にのみクリーンアップを実行する。
- **破棄の確認なし**
  - **問題**: 誤って作業を削除してしまう。
  - **修正**: オプション 4 では、タイプ入力による 'discard' の確認を求める。

### 危険な兆候 (Red Flags)

**決してしないこと：**
- テストが失敗したまま続行する。
- マージ結果のテスト検証を行わずにマージを完了する。
- 確認なしで作業を削除する。
- 明示的な要求なしに強制プッシュ（force-push）を行う。

**常にすること：**
- オプションを提示する前にテストを検証する。
- 正確に 4 つのオプションを提示する。
- オプション 4 ではタイプ入力による確認を得る。
- ワークツリーのクリーンアップはオプション 1 と 4 の場合のみ行う。


## 統合

**呼び出し元：**
- **subagent-driven-development** (ステップ 7) - すべてのタスク完了後
- **executing-plans** (ステップ 5) - すべてのバッチ完了後

**連携先：**
- **using-git-worktrees** - このスキルによって作成されたワークツリーをクリーンアップします。
