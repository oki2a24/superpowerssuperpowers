# 知見蒸留器 (Observation Distiller)

## 概要
セッション中の「失敗」「成功」「ユーザーの指摘」を、スキルや憲法を直接変更することなく、外部ファイル（Observations）として永続化・構造化するための規律です。

## いつ使用するか
- セッションを終了（リフレッシュ）する直前。
- 重要な設計上の指摘や、規律違反の注意を受けたとき。
- 大きな実装フェーズが完了したとき。

## 知見の地層と優先順位 (The Strata)
知見は以下の四層構造で管理され、**上位層（右側）が下位層（左側）を補足・上書き**し、最終的に AI の実行コンテキストへと合成されます。

```text
[ 低優先 ]                                                                                                          [ 高優先 ]
  L1: Core (Skills)             L2: Extension (Shared)                L3: Personal (Global)         L4: Project (Local)
  [Extension Root]/skills       [Extension Root]/observations         ~/.gemini/observations/       ./.gemini/observations/
          |                             |                             |                             |
          +-----------------------------+-----------------------------+-----------------------------+
                                        |
                                        v
                               [ AI Execution Context ]
```
## コアプロセス: RED-GREEN 知見採取

### 1. 課題の特定 (RED)
...
- 現在の AI がどのような「不適切な判断」や「規律違反」をしたか、物理的な証拠（ログ）から特定せよ。
- その「失敗」が、既存の知見がないために起きたのか、あるいは既存の知見が不十分なために起きたのかを明確にせよ。

### 2. 知見の記述とマージ (GREEN)
- **物理的なガードレールの策定**: 失敗を防ぐための具体的な手順、禁止事項、推奨ツールを記述せよ。
- **既存知見との論理的マージ**: 対象ファイルが既に存在する場合は、上書きせず既存知見と統合せよ。重複を排除し、AI が即座にアクションに繋げられる「物理的な指示」へと **抽象化** せよ。
- **保存先 (L2-L4) の決定**:
    - **L2 (Shared):** `[Extension Root]/observations/`。この拡張機能に同梱され、全ユーザーに共有されるベストプラクティス。
    - **L3 (Global):** `~/.gemini/observations/`。ユーザー個人の普遍的な流儀や好み。
    - **L4 (Project):** `./.gemini/observations/`。作業中のプロジェクト固有の局所的な知見。

### 3. 検証と洗練 (REFACTOR)
- 追加した知見を読み込んだ状態で、AI が正しく振る舞うか（失敗が再現しないか）を実証テストせよ。
- 曖昧な表現を排除し、AI が迷いなくアクションに繋げられる「物理的な指示」へと最終洗練せよ。

## 知見の仕分けガイドライン

| 保存先 | 典型的な内容 | 理由 |
| :--- | :--- | :--- |
| **Global** | 「AI は常に RED ステップを省略してはならない」 | 普遍的な行動原理であり、どのプロジェクトでも有効。 |
| **Local** | 「このプロジェクトでは Gemini CLI ネイティブタスクを優先せよ」 | プロジェクト固有のツール選定。 |
| **Local** | 「React コンポーネントは `src/components/ui/` に置く」 | プロジェクト固有の命名規則・構造。 |
| **Global** | 「`replace` ツールの `old_string` は多めに含める」 | ツールの使いこなしのコツ。 |

## 禁止事項
- **略称の使用**: ディレクトリやパスの記述において `obs` 等の略称を使用してはならない。必ず `observations`, `extensions` とフルスペルで記述せよ。
- **本体の汚染**: 知見を `GEMINI.md` や `SKILL.md` の本体（コード部分）に直接書き込んではならない。必ず `observations/` ディレクトリを使用せよ。
- **ログへの放置**: 重要な知見をチャットログの要約だけで終わらせてはならない。必ず物理的なファイルとして永続化せよ。

## よくある間違い
- **対象を特定しない**: どのスキルに対する知見か不明確な「雑記」として保存する。 -> 必ず対象名（スキル名等）と一致したファイル名にせよ。
