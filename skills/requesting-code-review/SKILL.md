# コードレビューの依頼 (Requesting Code Review)

`code-reviewer` サブエージェントを派遣し、問題が連鎖する前に修正します。レビュアーにはセッション履歴ではなく、評価のために精密に構成されたコンテキストが渡されます。これにより、レビュアーはあなたの思考プロセスではなく「成果物」に集中でき、あなた自身も作業を継続するためのコンテキストを維持できます。

**コア原則: 早く、頻繁にレビューする。**

## レビューを依頼するタイミング

**必須:**
- `subagent-driven-development` の各タスク完了後
- 大規模な機能の実装完了後
- `main` ブランチへのマージ前

**推奨 (価値のあるタイミング):**
- 行き詰まったとき（新しい視点を得るため）
- リファクタリング前（ベースラインの確認）
- 複雑なバグを修正した後

## 依頼の手順

**1. Git SHA（コミットハッシュ）の取得:**
レビュー対象となる変更の開始点と終了点を特定します。
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # または origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

**2. コードレビュアー・サブエージェントの派遣:**

`general-purpose` タイプのタスクツール（Gemini CLIでは `@generalist`）を使用し、`skills/requesting-code-review/code-reviewer.md` にあるテンプレートを埋めて派遣します。

**プレースホルダ:**
- `{DESCRIPTION}` - 何を構築したかの簡潔な要約。
- `{PLAN_OR_REQUIREMENTS}` - 本来どうあるべきか（計画ファイル、タスクテキスト、または要件）。
- `{BASE_SHA}` - 開始コミット。
- `{HEAD_SHA}` - 終了コミット。

**3. フィードバックへの対応:**
- **Critical (致命的)**: 直ちに修正が必要です。
- **Important (重要)**: 次のステップに進む前に修正すべきです。
- **Minor (軽微)**: 後で対応するためにメモしておきます。
- 反論: レビュアーの指摘が誤っていると思われる場合は、技術的な根拠（理由）を持って説明してください。

## 例 (Example)

```text
[タスク 2: 検証機能の追加 を完了]

あなた: 次に進む前に、コードレビューを依頼します。

BASE_SHA=$(git log --oneline | grep "Task 1" | head -1 | awk '{print $1}')
HEAD_SHA=$(git rev-parse HEAD)

[コードレビュアー・サブエージェントを派遣]
  DESCRIPTION: 4つの問題タイプを検知する verifyIndex() と repairIndex() を追加
  PLAN_OR_REQUIREMENTS: docs/plans/deployment-plan.md のタスク 2
  BASE_SHA: a7981ec
  HEAD_SHA: 3df7661

[サブエージェントの回答]:
  強み: クリーンなアーキテクチャ、実用的なテスト
  指摘事項:
    重要: 進捗インジケータが不足している
    軽微: 報告間隔にマジックナンバー (100) が使用されている
  評価: 修正後に進めて良い

あなた: [進捗インジケータを修正]
[タスク 3 へ進む]
```

## ワークフローとの統合

**サブエージェント駆動開発 (SDD):**
- 各タスクの完了直後にレビューを行い、問題が連鎖するのを防ぎます。
- 修正を完了してから次のタスクへ進みます。

**計画の実行 (Executing Plans):**
- 各タスクの完了時、または自然なチェックポイントでレビューを受けます。
- フィードバックを適用してから続行します。

**アドホックな開発:**
- マージ前に必ずレビューを行います。
- 作業に行き詰まったときに依頼します。

## レッドフラッグ (Red Flags)

**絶対に行わないこと:**
- 「簡単だから」という理由でレビューをスキップする。
- Critical（致命的）な問題を無視する。
- Important（重要）な問題を修正せずに次へ進む。
- 正当な技術的フィードバックに対して感情的に反論する。

**レビュアーの指摘が誤っていると思われる場合:**
- 技術的な根拠（理由）を持って反論してください。
- 動作を証明するコードやテストを提示してください。
- 不明点がある場合は、明確化を求めてください。
