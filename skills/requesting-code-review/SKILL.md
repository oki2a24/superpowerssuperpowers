# コードレビューの依頼 (Requesting Code Review)

`code-reviewer` サブエージェントを呼び出し、実装内容が計画や品質基準に適合しているかを検証します。

**コア原則:** 「早く、頻繁にレビューする」。問題が蓄積する前に修正します。

## レビューを依頼するタイミング

**必須:**
- `subagent-driven-development` の各タスク完了後
- 大規模な機能の実装完了後
- `main` ブランチへのマージ前

**推奨:**
- 行き詰まったとき（新しい視点を得るため）
- リファクタリング前（ベースラインの確認）
- 複雑なバグを修正した後

## 依頼の手順

**1. 変更範囲（Git SHA）の特定:**
レビュー対象となる変更の開始点と終了点を特定します。
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # または main
HEAD_SHA=$(git rev-parse HEAD)
```

**2. `subagent` ツールの呼び出し:**

`subagent` ツールを使用し、役割として `code-reviewer` を指定（またはそのプロンプトを渡す）して、以下の情報を `objective` に含めます。

- **実装内容 (`WHAT_WAS_IMPLEMENTED`)**: 今回構築したもの。
- **計画・要件 (`PLAN_OR_REQUIREMENTS`)**: 本来どうあるべきか（計画書への参照など）。
- **変更範囲 (`BASE_SHA`...`HEAD_SHA`)**: レビュー対象のコミット。
- **概要 (`DESCRIPTION`)**: 変更の簡潔な要約。

**3. フィードバックへの対応:**
- **Critical (致命的)**: 直ちに修正が必要です。
- **Important (重要)**: 次のステップに進む前に修正すべきです。
- **Suggestions (提案/軽微)**: 必要に応じて修正するか、将来の改善として記録します。
- **反論**: レビュアーの指摘が誤っていると思われる場合は、技術的な根拠を持って説明し、必要に応じてコードやテストで証明してください。

## ワークフローとの統合

**サブエージェント駆動開発 (SDD):**
- 各タスクの後にレビューを行い、問題が連鎖するのを防ぎます。
- 修正してから次のタスクへ進みます。

**計画の実行 (Executing Plans):**
- 数個のタスク（バッチ）ごとにレビューを受けます。

## 禁止事項

- 「簡単だから」という理由でレビューをスキップすること。
- Critical な問題を無視すること。
- 指摘に対して感情的に反論すること。
