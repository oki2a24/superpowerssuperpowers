## レビュー・チェックリスト

**コード品質:**
- 関心の分離（Separation of Concerns）が明確か？
- 適切なエラーハンドリングが行われているか？
- 型安全性（適用可能な場合）は保たれているか？
- DRY 原則に従っているか？
- エッジケースは考慮されているか？

**アーキテクチャ:**
- 設計上の決定は妥当か？
- 拡張性（Scalability）は考慮されているか？
- パフォーマンスへの影響は検討されているか？
- セキュリティ上の懸念はないか？

**テスト:**
- テストは実際にロジックをテストしているか（モックに頼りすぎていないか）？
- エッジケースはカバーされているか？
- 必要に応じて結合テストが含まれているか？
- すべてのテストがパスしているか？

**要件:**
- 計画のすべての要件を満たしているか？
- 実装は仕様と一致しているか？
- スコープクリープ（不要な機能追加）はないか？
- 破壊的変更はドキュメント化されているか？

**本番準備状況:**
- 移行戦略（スキーマ変更がある場合）はあるか？
- 後方互換性は考慮されているか？
- ドキュメントは完備されているか？
- 明らかなバグはないか？

## 重要なルール

**実行すべきこと:**
- 実際の重要度で分類する（すべてを Critical にしない）。
- 具体的に記述する（ファイル名:行番号を明記）。
- なぜその問題が重要なのかを説明する。
- 強みを認める。
- 明確な判定を下す。

**避けるべきこと:**
- 内容をチェックせずに「問題なし」と言う。
- 重箱の隅をつつくような指摘（Nitpick）を Critical にする。
- レビューしていないコードに対してフィードバックする。
- 曖昧な表現（「エラーハンドリングを改善してください」など）を使う。
- 明確な判定を避ける。
