# ロードマップ管理 (Roadmap Management)

## 概要
長期的な開発セッションにおける「現在地」と「目標」を常に明確にし、エージェントの脱線とユーザーの不安を解消するための規律です。

## いつ使用するか
- 3つ以上の独立したタスクを含む作業を開始するとき。
- 複数のスキル（例：`brainstorming` -> `writing-plans` -> `TDD`）を跨いで使用するとき。
- セッションが 10ターンを超えると予想されるとき。

## コアパターン：フルレポーティング
大規模タスクの開始時およびタスクの変わり目（コミット直後）に、以下の形式で進捗を同期します。

### 1. セッション開始時：ロードマップの提示
```markdown
### **本セッションのロードマップ**
1.  **Task 1: [名称]** (現在地)
    - **Skill**: [使用予定のスキル]
    - **GREEN**: [完了とみなす具体的な状態]
2.  **Task 2: [名称]**
    - **Skill**: [使用予定のスキル]
    - **GREEN**: [完了とみなす具体的な状態]
...
N.  **Task N: [名称]**
```

### 2. タスク間：次タスクの完全な再掲
コミット直後、または大きなステップの完了時に、次のタスクの詳細を**一切省略せずに**報告します。

```markdown
**現在地：Task M 完了**

**次：Task M+1 [名称]**
- **Skill**: `[使用予定のスキル]`
- **GREEN**: [完了とみなす具体的な状態]
```

## 実装（エージェントへの指示）
- **必ず GREEN 条件を定義せよ**: 曖昧な「実装する」ではなく、「テストがパスし、ファイル X が更新されること」のように具体的に記述せよ。
- **スキルの宣言を義務化せよ**: 各タスクでどのスキル（規律）に従うかをあらかじめユーザーと合意せよ。
- **省略の禁止**: 「Skill は前回提示したから省略する」といった自己判断は禁止します。各ターンの終わりに、ユーザーと私の「次に守るべき規律」の合意を物理的に形成してください。

## よくある間違い
- **ロードマップが一度きり**: 最初に出したきり、更新されない。 -> タスクごとに「現在地」を報告せよ。
- **GREEN 条件が不明確**: 何をもってそのタスクが「終わり」なのか分からない。 -> 具体的・物理的な証拠を定義せよ。
- **計画の乖離（Drifting）**: 作業途中で当初のタスク順序や内容が変わったのに、古いロードマップに従い続ける。 -> 逸脱を検知した瞬間、直ちにロードマップを修正・再提示し、現在の実態と同期せよ。
- **情報の圧縮**: 次のタスクを「次：Task 2」のように省略して報告する。 -> ユーザーに不安を与えるため禁止。必ずフルレポートせよ。
