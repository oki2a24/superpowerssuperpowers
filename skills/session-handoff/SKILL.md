# 🔄 Session Handoff

## 概要
セッションの肥大化やコンテキストの切り替えが発生した際、現在の思考状態、物理的な作業状況、および「なぜその結論に至ったか」という背景を構造化し、次のセッションへ無損失で引き継ぐためのコアスキルです。

## いつ使用するか
- ユーザーが「セッションを切り替えたい」「別の話題へ」といった意図を示したとき。
- トークン量が増え、AIの反応が遅くなったり誤解が生じ始めたとき。
- 予期せぬ中断が発生し、作業状態を「スナップショット」として保存したいとき。

## コアパターン（ハンドオーバー・メモ）

AIは以下のテンプレートに従い、**情報を一切圧縮せず**に出力しなければなりません。

```markdown
# 🔄 Session Handoff: [タスク名/トピック]

## 🎯 最終目標 (Ultimate Goal)
- このセッション全体で達成しようとしていたこと。

## ✅ 完了した事項と「意思決定の背景」 (Done & Why)
- [済] 〇〇の実装/調査
  - **Why:** なぜこのアプローチを選んだか。試行して失敗した案は何か。
  - **Crucial:** 「自明だから」という理由で背景を省略してはならない。

## 🚧 現在の物理的状態 (Physical Anchor)
- **作業中のファイル:** `path/to/file.ts` (L100付近)
- **直近の成功コマンド:** `npm test`
- **直近の失敗/未検証:** `npm run lint` (エラーメッセージ: ...)
- **Snapshot:** 現在の `todo.mjs` の最新状態。

## 📝 次の具体的なアクション (Next Steps)
1. `path/to/file.ts` を読み込む。
2. 〇〇のロジックを修正する。

## 💬 再開用プロンプト (Resumption Prompt)
> [次のセッションのAIが即座に起動し、迷わず最初のコマンドを叩ける具体的な依頼文]
```

## 正当化（言い訳）の封鎖

| AIの言い訳 | 現実（規律） |
| :--- | :--- |
| 「詳細は設計ドキュメントにあるから省略した」 | 設計ドキュメントは「経緯」であり、ハンドオーバーは「現在の思考の鮮度」である。省略不可。 |
| 「AIなら `ls` で探せばいいからファイルパスは不要」 | 無駄な探索はトークンの浪費と誤解の元。物理的アンカーを明記せよ。 |
| 「文脈から自明なことは書かなくてよい」 | その「自明」の欠落が再開時の誤解を生む。背景こそが情報の本体である。 |
| 「要約した方がトークン効率が良い」 | 効率よりも「再現性」と「正確性」を優先せよ。不正確な再開は最大の浪費である。 |

## レッドフラグ - 停止してやり直す
- 「～を完了しました」という一言だけの報告（Why が欠落している）。
- 具体的なファイルパスや行番号が含まれていない。
- 次のセッションのAIが最初に行うべき具体的なコマンド（`npm test` 等）が書かれていない。

## 鉄の掟
**「自明だから」はAIの怠惰である。背景と物理的アンカーを物理的に記述せよ。**
