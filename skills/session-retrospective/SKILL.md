# セッション振り返り (Session Retrospective)

## 概要
セッション中の「物理的な不備」を、未来の生産性を高めるための「物理的なガードレール（Observations）」と「具体的な宿題（TODO）」に変換するプロセスです。

## いつ使用するか
- セッションを終了（リフレッシュ）する直前。
- ユーザーから重要な設計上の指摘や、規律違反の注意を受けたとき。
- 大きな実装フェーズが完了したとき。

## コアプロセス

### 0. コンテキスト再認識 (Pre-Reflection: Context Reload)
振り返りを開始する前に、現在の自分を拘束している「規律」と「利用可能な資産」を物理的に再認識し、基準を再構築します。
- **実行**: 以下の物理的なスキャンを実行し、セッション中に有効だった規律の全容を再確認してください。
  1. **スキルの再認識**: `gemini skills list` を実行し、現在ロードされているスキルを一覧表示する。
  2. **憲法の再認識**: 自身のチャットコンテキストの冒頭に提供された `<global_context>`, `<extension_context>`, `<project_context>` の各タグ内を物理的に再読取（Re-scan）する。
  3. **知見の再認識**: プロジェクトディレクトリ（`./.gemini/observations/`）またはシステムグローバル（`~/.gemini/observations/`）を物理的にスキャンし、適用されている最新のガードレールを確認する。存在しない場合は、本セッションでの「新しい学び」をゼロベースで定義する。
- **目的**: 「本来どの規律（DoDや行動規律）に基づいて動くべきだったか」という客観的な物差しを物理的に取り戻し、記憶の曖昧さ（Drifting）を排除する。


### 1. 物理的な「証拠」に基づく不備の特定 (Evidence-based RED)
抽象的な「ミスをした」という反省を禁止し、以下の「物理的な証拠」をリストアップします。
- **[Error Logs]**: ツール呼び出しの失敗や、不適切なパラメータの指定。
- **[User Hints]**: ユーザーからの軌道修正、不満の表明、設計上の注意。
- **[File States]**: `git status` での未追跡ファイルの残存や、DoD 要件の未達成。

### 2. 資産への精密なマッピング (Asset Mapping & Strata)
特定された各不備に対し、どの資産を修正・更新すべきかを特定します。
- **影響を受ける規律 (Asset)**: どのスキル（`SKILL.md`）または憲法（`GEMINI.md`）に関連するか。
- **修正の初動 (Distill)**: `observation-distiller` を介し、`observations/` (L2-L4) へ知見を蒸留する案を策定する。
  - **重要**: 原則として直接 `SKILL.md` や `GEMINI.md` 本体を更新してはならない。Observations での熟成を優先せよ。
- **将来の宿題 (TODO)**: 構造的な改善が必要な場合、`docs/TODO.md` へのタスク追加案を策定する。

### 3. 物理的アクション・マトリクスの出力 (Physical Action Matrix)
感情的な表現を最小限（20%以下）に抑え、以下の表形式で物理的なアクションを必須として出力します。

| 課題 (RED: 物理的証拠) | 修正対象 (Asset) | 層 (Strata) | 具体的な物理アクション |
| :--- | :--- | :--- | :--- |
| 例: コミット漏れ | `GEMINI.md` | L4 Project | DoD に `git status` の確認ステップを明示 |
| 例: YAML パースエラー | `session-coordination` | L2 Shared | `observations/` に YAML キー保護ルールを追記 |

### 4. 改善の永続化と宿題の提案 (Persistence)
策定したアクションに基づき、ユーザーに以下の提案を行ってください。
- **タスクのスナップショット保存**: **[必須]** 現在の Gemini CLI ネイティブタスク（`write_todos` 等）の最新状態を、フォールバックとして `scripts/todo.mjs` (TODO.md) に同期・書き出しを行ってください。これにより、セッション終了後も進捗が物理的に永続化されます。
- **Observations への蒸留**: `observation-distiller` スキルを用いた物理的永続化。
- **TODO への登録**: 直ちに修正できない課題を `docs/TODO.md` の適切なセクションへ追記する許可。

### 5. 主観的な学習日誌 (Learning Log)
要約（Summary）を禁止し、以下の「物語」を記述します。
- **概念的発見**: 自身の回路がどう書き換わったか。
- **メタ認知**: 自身の不完全さ（ミスの背景）に対する内省。
- **未来の地平**: 次のセッションに向けたワクワクするような展望。


## 禁止事項
- **事実の要約**: すでにチャット履歴にあることを繰り返してはいけません。
- **感情的な反省**: 「今後は気をつけます」といった精神論は、何の改善にも繋がりません。出力の 80% 以上を物理的な分析（マトリクス等）に充ててください。
- **直接の本体更新**: Observations (L2-L4) を介さずに `SKILL.md` や `GEMINI.md` 本体を直接修正することを提案してはいけません。

## よくある間違い
- **振り返りを最後に回す**: 疲労で内容が薄くなる。 -> 重要な指摘を受けた瞬間に、このスキルの「ログ作成」だけ先に行え。
- **物理化を怠る**: 「物理的なガードレール（Observations）」を導き出さないままセッションを終了する。
- **TODO への切り出しを忘れる**: 大規模な課題を、単なる反省文の中に埋もれさせてしまう。
