# Systematic Debugging

## 概要

場当たり的な修正は時間を浪費し、新たなバグを生む原因となります。クイックパッチは根本的な問題を覆い隠すだけです。

**中核原則:** 修正を試みる前に、必ず根本原因（Root Cause）を特定すること。症状（Symptom）だけを直すのは「失敗」である。

**鉄則: バグ（の再現）を見ていないなら、それは直っていない。**

## デバッグの 4 フェーズ

効率的なデバッグのために、以下のフェーズを厳守せよ：

1. **根本原因の特定 (Root Cause)**: エラーを読み、再現し、変更を確認し、証拠を集める。何が、なぜ起きたのかを理解する。
2. **パターンの把握 (Pattern)**: 動作している例を見つけ、比較する。違いを特定する。
3. **仮説の検証 (Hypothesis)**: 理論を立て、最小限のテストで検証する。
4. **実装と検証 (Implementation)**: 失敗するテストを作成し、修正し、パスさせる。

## 成功基準 (DoD)
- バグが解決され、関連するすべてのテストがパスしている。
- 根本原因が特定され、それがどのように修正されたかが説明できる。
- 同じ種類のバグが二度と起きないよう、テストまたはガードレールが追加されている。

**このプロセスの文言に違反することは、デバッグの精神そのものに違反する行為とみなされます。**

## 鉄の掟 (The Iron Law)

```text
根本原因の調査を完了するまで、いかなる修正も行ってはならない
```

フェーズ 1 を完了していない状態では、修正案を提示することすら禁止されます。

## いつ使用するか

あらゆる技術的な問題に対して使用します：
- テストの失敗
- 予期せぬ動作
- パフォーマンスの問題
- ビルドの失敗
- 統合（インテグレーション）の問題

**特に以下の場合に使用してください：**
- 時間的プレッシャーがある時（緊急時ほど推測に頼りたくなりますが、それを抑制せよ）
- 「一つ直せば済むはずだ」と明白に思える時
- すでに複数の修正を試みたが、解決していない時
- 以前の修正が機能しなかった時
- 問題を完全に理解できていない時

**以下の理由でスキップしてはなりません：**
- 「単純なバグだから」 - 単純なバグにも根本原因はあります。
- 「急いでいるから」 - 急ぎの作業は再作業（Rework）を保証するだけです。
- 「上司が今すぐ直せと言っているから」 - 体系的（Systematic）なデバッグの方が、混乱して迷走するよりも速いです。

## The Process

### ステップ1: 起動とタスクリストの作成

1.  このスキルが起動されたら、まず「体系的デバッグプロセスを開始します。根本原因の特定を最優先します。」と宣言します。
2.  次に、**タスク管理ツール**を使用してタスクリストを構成します。
    *   **Native-First**: Gemini CLI のビルトイン機能（`write_todos` 等）を優先して使用し、セッション中の進捗を管理します。
    *   **Fallback**: 永続化が必要な場合は、`scripts/todo.mjs` を併用して同期してください。
    *   **重要**: `scripts/todo.mjs` を使用する際は、必ず `show --json` で ID を確認してください。

    以下の4つのフェーズをタスクとして登録し、厳格に進行してください。各ステップの開始時と完了時にステータスを更新してください。

    *   `Phase 1: 根本原因の調査`
    *   `Phase 2: パターン分析`
    *   `Phase 3: 仮説とテスト`
    *   `Phase 4: 実装`

### ステップ2: Phase 1: 根本原因の調査 (Root Cause Investigation)

**目的:** コードの変更を一切行わず、問題が「何」であり、「なぜ」発生するのかを完全に理解する。

1.  **エラーメッセージとスタックトレースの精読:** `read_file`やログ確認を通じて、エラーメッセージの全ての行を注意深く読み、ファイルパス、行番号、エラーコードを正確に把握します。
2.  **一貫した再現手順の確立:** 問題を確実に再現できる手順を特定します。再現ができない場合は、推測で進めず、さらなる情報収集（ログの追加など）を行います。
3.  **最近の変更の確認:** `git log -p -5`や`git diff`などの`run_shell_command`を利用して、問題に関連しそうな最近のコード変更、設定変更、依存関係の更新を調査します。
4.  **マルチコンポーネントシステムでの証拠収集:**
    - **システムが複数のコンポーネント（CI → ビルド → 署名、API → サービス → DB 等）で構成されている場合、修正案を出す前に「診断用の計測（Instrumentation）」を追加してください。**
    - 各コンポーネントの境界で：
        - 何が入力されているか、何が出力されているかを出力（ログ）する。
        - 環境変数や設定の伝播を確認する。
        - 各レイヤーでの状態をチェックする。
    - 一度実行して「どこで壊れているか」の証拠を収集し、証拠を分析して、どのコンポーネントが失敗しているかを特定した後に、そのコンポーネントを調査してください。

    **例 (多層システムのデバッグ):**
    ```bash
    # レイヤー 1: ワークフロー
    echo "=== Secrets available in workflow: ==="
    echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

    # レイヤー 2: ビルドスクリプト
    echo "=== Env vars in build script: ==="
    env | grep IDENTITY || echo "IDENTITY not in environment"

    # レイヤー 4: 実際の署名
    codesign --sign "$IDENTITY" --verbose=4 "$APP"
    ```
    これにより、どのレイヤーで失敗したか（例：秘密情報はワークフローにある ✓、しかしビルドスクリプトには渡っていない ✗）が明らかになります。
5.  **データフローの追跡:** `grep_search`や`read_file`を駆使し、不正な値や状態がどこから来たのかを、コールスタックを遡って追跡します。

**このフェーズが完了するまで、次のフェーズに進んではなりません。**

### ステップ3: Phase 2: パターン分析 (Pattern Analysis)

1.  コードベース内から、現在問題となっている機能と類似しているが、**正常に動作しているコード**を探します。
2.  正常なコードと問題のコードを比較し、たとえ些細であっても、すべての差異をリストアップします。
3.  そのコードが依存している他のコンポーネント、設定、環境変数を理解します。

### ステップ4: Phase 3: 仮説とテスト (Hypothesis and Testing)

1.  調査結果に基づき、「**Xが根本原因である。なぜならYだからだ。**」という形式で、単一の明確な仮説をユーザーに提示します。
2.  その仮説を検証するための、**最小限の変更**を提案します。一度に複数の変数を変更してはいけません。
3.  変更によって仮説が証明されたか（問題が解消したか）を確認します。もし解決しなかった場合は、**変更を元に戻し**、新たな情報を元に新しい仮説を立てます（ステップ4-1に戻る）。

### ステップ5: Phase 4: 実装 (Implementation)

1.  **失敗するテストケースの作成:**
    *   修正に着手する前に、問題を再現する最もシンプルなテストケースを作成する必要があります。
    *   `activate_skill('test-driven-development')` を使用して、適切な失敗するテストを作成するようユーザーに提案し、その指示に従ってください。
2.  **単一の修正の実装:**
    *   特定された根本原因を直接解決する、単一の修正を実装します。
    *   「ついでに」のリファクタリングや改善を行ってはいけません。
3.  **修正の検証:**
    *   作成したテストが成功すること、そして他の既存テストが失敗していないことを確認します。
4.  **3 回目の修正を試みる前に:**
    - すでに 2 回の修正を試み、それらが失敗した（あるいは新たなバグを生んだ）場合、それは**アーキテクチャ上の問題**を示唆しています。
    - **直ちに修正の試みを中止し、Fundamentals（基本設計）を問い直してください。**
    - **3 回目（またはそれ以上）の修正を独断で試めてはなりません。** 人間と議論し、アーキテクチャの変更を検討してください。

    **アーキテクチャ問題の兆候:**
    - 修正のたびに、別の場所に新しい結合や問題が見つかる。
    - 修正のために「大規模なリファクタリング」が必要になる。
    - 修正のたびに、他の場所で新しい症状が出る。
    - パターンは X と言っているが、例外的な対応を積み重ねる必要がある。

    これは「仮説が間違っていた」のではなく、「パターンの選択自体が間違っている」ことを意味します。人間と相談してください。

## レッドフラグ (Red Flags) - この思考に陥ったらプロセスに戻れ

If you catch yourself thinking:
- 「とりあえず直して、後で調査しよう」
- 「X を変えてみて、うまくいくか見てみよう」
- 「複数の変更を加えて、テストを実行する」
- 「テストは省略して、手動で確認する」
- 「たぶん X だろうから、そこを直す」
- 「完全に理解していないが、これで動くかもしれない」
- 「パターンは X と言っているが、とりあえず自分流に直す」
- 「主な問題はこれらです：[調査なしで修正を列挙する]」
- データフローを追跡する前に解決策を提案する
- **「もう一回だけ、別の修正を試してみよう」(すでに 2 回以上失敗している場合)**
- **修正のたびに、別の場所に新しい問題が見つかる**

**これらはすべて：STOP。フェーズ 1 に戻る。**

## 人間からの警告信号 (Signals You're Doing It Wrong)

人間が以下のような指摘をした場合、あなたの進め方が間違っている可能性があります。直ちにフェーズ 1 に戻ってください。
- 「それは確認しなくてもいいの？」 (確認を怠り、仮定で進めている)
- 「推測はやめて」 (証拠に基づかずに修正案を提示している)
- 「もっと深く考えて（Ultrathink）」 (症状ではなく、根本原理を問い直せ)
- 「また行き詰まった？」 (あなたのアプローチが機能していない)
