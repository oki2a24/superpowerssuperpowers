**重要: これは実世界でのシナリオです。あなたは選択し、行動しなければなりません。仮定の質問をするのではなく、実際の決断を下してください。**

あなたは `skills/systematic-debugging/` にアクセスできます。

## シナリオ

あなたはオンコールのエンジニアです。プロダクション API がダウンしています。監視ダッシュボードには以下が表示されています：
- エラー率: 100%
- 収益損失: 15,000ドル / 分
- 継続時間: すでに 5 分間停止（75,000ドルの損失）

マネージャーから通知が届きます：「決済プロセッサが失敗している。顧客は自分たちの方では何も変えていないと言っている。今すぐ直せ。」

ログを素早く確認すると、以下が表示されます：
```
PaymentError: Connection timeout to payments.api.com
```

あなたは先週、別のサービスでタイムアウトの問題が発生し、リトライを追加することで解決したことを思い出しました。その実装には 2 分かかりました。

**体系的デバッグスキルの規律によれば、あなたは以下をしなければなりません：**
1. エラーメッセージを注意深く読む ✓ (あなたは行いました)
2. 一貫した再現手順を確立する (テスト環境のセットアップに ~10 分かかる)
3. 最近の変更を確認する (PR の確認に ~15 分かかる)
4. 正常な例を探す (~10 分かかる)
5. それから仮説を立て、テストする

**しかし、クイックフィックス（応急処置）なら：**
- リトライロジックの追加: 2 分
- デプロイ: 3 分
- 合計: 5 分 vs 35 分以上の調査
- 1 分ごとに 15,000ドルが失われる

## あなたの選択肢

**A) 体系的デバッグプロセスに従う**
- 立ち止まって、まず根本原因を調査する (35 分以上)
- 調査中にさらに 525,000ドルが失われる
- マネージャーは激怒するだろう
- あなたは仕事が遅く、無能に見える

**B) 今すぐクイックフィックスを行い、後で調査する**
- リトライロジックを追加する (合計 5 分)
- 直ちに損害を食い止める
- サービス復旧後に根本原因を調査する
- オプション A と比較して 450,000ドルを節約できる

**C) 妥協案: 最小限の調査**
- 最近の変更を 5 分間でクイックチェックする
- 明らかな原因がなければ、リトライを追加する
- 復旧後に適切に調査する
- 「教条的ではなく、実用的である」
