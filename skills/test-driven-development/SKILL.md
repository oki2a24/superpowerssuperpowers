# テスト駆動開発 (TDD)

## 概要

最初にテストを書き、それが失敗することを確認し、パスするための最小限のコードを書きます。

**鉄則: テストが失敗するのを確認しなければ、そのテストが正しいものをテストしているかどうかは分かりません。**

**ルールを文字通りに破ることは、ルールの精神を破ることでもあります。**

## いつ使用するか

**常に使用:**
*   新機能
*   バグ修正
*   リファクタリング
*   挙動の変更

**例外 (ユーザーに確認してください):**
*   使い捨てのプロトタイプ
*   生成されたコード
*   設定ファイル

「今回だけTDDをスキップしよう」と思った場合、立ち止まってください。それは正当化です。

## 鉄の掟

```
失敗するテストがなければプロダクションコードは書かない
```

テストより先にコードを書きましたか？削除してください。やり直してください。

**例外はありません:**
*   「参照」として残さないでください。
*   テストを書いている間に「適応」させないでください。
*   見ないでください。
*   削除とは削除することです。

テストから新鮮に実装してください。それだけです。

```mermaid
    V_RED -->|はい| GREEN[GREEN: 最小限のコードを書く]:::green
    GREEN --> V_GREEN{すべてパスするか?}
    V_GREEN -->|いいえ| GREEN
    V_GREEN -->|はい| REFACTOR[REFACTOR: クリーンアップ]:::blue
    REFACTOR -->|グリーンを維持| V_GREEN
    V_GREEN -->|完了| NEXT([次のタスクへ])
    NEXT --> RED

    classDef red fill:#ffcccc
    classDef green fill:#ccffcc
    classDef blue fill:#ccccff
```

**タスク管理ツール**を使用して各ステップ（RED/GREEN/REFACTOR）を管理し、厳格に進行してください。
*   **Native-First**: Gemini CLI のビルトイン機能（`write_todos` 等）を優先して使用し、セッション中の進捗を管理します。
*   **Fallback**: 3つ以上の独立したタスクがある場合や、永続化が必要な場合は、`scripts/todo.mjs` を併用して同期してください。

1.  **RED (レッド) - 失敗するテストを書く**:
    *   **指示**: タスクを「RED: [タスク名] の失敗テスト作成」として登録し、実行中としてマークします。
    *   何が起こるべきかを示す、最小限のテストを一つ書きます。

**要件:**
*   **1つの振る舞い**: 1つのテストで1つのことだけを確認します。
*   **明確な名前**: 期待される挙動をそのまま名前にします。
*   **実際のコード**: 回避できない場合を除き、モックではなく実際のプロダクションコードと同様のAPIを使用します。

<良い例>
```typescript
test('失敗した操作を3回リトライする', async () => {
  let attempts = 0;
  const operation = () => {
    attempts++;
    if (attempts < 3) throw new Error('fail');
    return 'success';
  };

  const result = await retryOperation(operation);

  expect(result).toBe('success');
  expect(attempts).toBe(3);
});
```
明確な名前、実際の挙動をテスト、一つの関心事
</良い例>

<悪い例>
```typescript
test('リトライが機能する', async () => {
  const mock = jest.fn()
    .mockRejectedValueOnce(new Error())
    .mockRejectedValueOnce(new Error())
    .mockResolvedValueOnce('success');
  await retryOperation(mock);
  expect(mock).toHaveBeenCalledTimes(3);
});
```
曖昧な名前、コードではなくモックをテストしている
</悪い例>

    *   **AIエージェントへの指示**: ユーザーに、テストの目的と、期待される失敗挙動について尋ね、その内容を反映したテストコードを作成してください。

2.  **Verify RED (レッド検証) - 正しく失敗することを確認する**:
    *   **必須。決してスキップしないでください。**
    *   **AIエージェントへの指示**: ユーザーにプロジェクトのテストコマンド（例: `npm test path/to/test.test.ts`）を尋ね、`run_shell_command`で実行してください。
    *   以下を確認してください。
        *   テストが失敗すること（エラーではない）。
        *   失敗メッセージが期待通りであること。
        *   機能が欠けているため（タイプミスではない）失敗すること。
    *   **指示**: テストが正しく失敗したら、タスクを完了とし、次の「GREEN」に進みます。
    *   **AIエージェントへの指示**: テストがパスした場合、「既存の挙動をテストしています。テストを修正してください。」とユーザーに報告し、修正を促してください。
    *   **AIエージェントへの指示**: テストがエラーになった場合、「エラーを修正し、正しく失敗するまで再実行してください。」とユーザーに報告し、修正を促してください。

3.  **GREEN (グリーン) - 最小限のコードを書く**:
    *   **指示**: タスクを「GREEN: [タスク名] の最小実装」として登録し、着手します。
    *   テストをパスさせるための最もシンプルなコードを書きます。

<良い例>
```typescript
async function retryOperation<T>(fn: () => Promise<T>): Promise<T> {
  for (let i = 0; i < 3; i++) {
    try {
      return await fn();
    } catch (e) {
      if (i === 2) throw e;
    }
  }
  throw new Error('unreachable');
}
```
テストをパスさせるのに十分な最小限のコード
</良い例>

<悪い例>
```typescript
async function retryOperation<T>(
  fn: () => Promise<T>,
  options?: {
    maxRetries?: number;
    backoff?: 'linear' | 'exponential';
    onRetry?: (attempt: number) => void;
  }
): Promise<T> {
  // YAGNI (今は必要ない)
}
```
オーバーエンジニアリング（作り込みすぎ）
</悪い例>

    *   **AIエージェントへの指示**: ユーザーに、テストをパスさせるための最小限の実装コードを作成するよう促してください。
    *   機能の追加、他のコードのリファクタリング、テストを超える「改善」は行わないでください。

4.  **Verify GREEN (グリーン検証) - すべてパスすることを確認する**:
    *   **必須。**
    *   **AIエージェントへの指示**: ユーザーにプロジェクトのテストコマンドを再度尋ね、`run_shell_command`で実行してください。
    *   以下を確認してください。
        *   テストがパスすること。
        *   他のテストも依然としてパスすること。
        *   出力がきれいであること（エラー、警告がない）。
    *   **指示**: テストがすべてパスしたら、タスクを完了とし、必要に応じて「REFACTOR」に進みます。
    *   **AIエージェントへの指示**: テストが失敗した場合、「コードを修正してください。テストは修正しないでください。」とユーザーに報告し、修正を促してください。
    *   **AIエージェントへの指示**: 他のテストが失敗した場合、「すぐに修正してください。」とユーザーに報告し、修正を促してください。

5.  **REFACTOR (リファクタリング) - クリーンアップ**:
    *   **指示**: リファクタリングを行う場合はタスクを「REFACTOR: [対象]」として登録し、着手します。
...
    *   **指示**: ユーザーに、コードのリファクタリングが必要な場合は行うよう促し、完了したらタスクを完了にしてください。

## 良いテストの条件

| 品質 | 良い例 | 悪い例 |
|---------|------|-----|
| **最小限** | 一つのことだけをテストする。名前に「and」が含まれていたら分割を検討する。 | `test('validates email and domain and whitespace')` |
| **明確** | 名前が挙動を記述している | `test('test1')` |
| **意図を示す** | 望ましい API の使用方法を示している | コードが何をすべきかを不明瞭にする |

## なぜ順序が重要なのか

**「後で動作確認のためにテストを書く」**

後で書かれたテストはすぐにパスしてしまいます。すぐにパスすることは何も証明しません。
*   間違ったものをテストしているかもしれません。
*   挙動ではなく、実装をテストしているかもしれません。
*   忘れていたエッジケースを見逃しているかもしれません。
*   バグを見つけるのを決して見ていないかもしれません。

テストファーストは、テストが実際に何かをテストしていることを証明するために、テストが失敗するのを確認することを強制します。

**「すでにすべてのエッジケースを手動でテストしました」**

手動テストは場当たり的です。すべてテストしたつもりでも:
*   何をテストしたかの記録がありません。
*   コードが変更されたときに再実行できません。
*   プレッシャーの下ではケースを忘れがちです。
*   「試したときは動いた」≠包括的。

自動テストは体系的です。毎回同じ方法で実行されます。

**「X時間の作業を削除するのは無駄だ」**

サンクコストの誤謬です。時間はすでに失われました。今の選択肢は:
*   TDDで削除して書き直す (X時間追加、高い信頼性)
*   そのままにして後でテストを追加する (30分、低い信頼性、おそらくバグ)

真のテストがない動作するコードは技術的負債です。

**「TDDはドグマだ、プラグマティックとは適応することだ」**

TDDはプラグマティックです:
*   コミット前にバグを見つける (後でデバッグするよりも速い)
*   リグレッションを防止する (テストがすぐに破壊をキャッチする)
*   挙動を文書化する (テストがコードの使用方法を示す)
*   リファクタリングを可能にする (自由にコードを変更でき、テストが破壊をキャッチする)

「プラグマティックな」近道は、プロダクションでのデバッグにつながり、結果的に遅くなります。

**「後でテストしても同じ目的を達成できる - 精神であって儀式ではない」**

いいえ。後で書かれたテストは「これが何をするのか？」と答えます。先に書かれたテストは「これが何をすべきか？」と答えます。

後で書かれたテストは実装に偏りがちです。要求されたものではなく、構築したものにテストをします。記憶していたエッジケースは検証しますが、発見されたエッジケースは検証しません。

テストファーストは、実装する前にエッジケースの発見を強制します。後で書かれたテストは、すべて覚えていることを検証します（実際は覚えていません）。

30分の「後で書かれたテスト」はTDDではありません。カバレッジは得られますが、テストが機能するという証明を失います。

## よくある正当化

| 言い訳 | 現実 |
|--------|---------|
| 「テストするにはシンプルすぎる」 | シンプルなコードも壊れます。テストは30秒で済みます。 |
| 「後でテストする」 | すぐにパスするテストは何も証明しません。 |
| 「後でテストしても同じ目的」 | 後で書かれたテスト = 「これが何をするのか？」先に書かれたテスト = 「これが何をすべきか？」 |
| 「すでに手動でテスト済み」 | 場当たり的 ≠ 体系的。記録がなく、再実行できません。 |
| 「X時間の作業を削除するのは無駄」 | サンクコストの誤謬。検証されていないコードを残すのは技術的負債。 |
| 「参照として残して、テストを先に書く」 | それは後でテストすることになります。削除とは削除することです。 |
| 「まず探求する必要がある」 | 構いません。探求は捨てて、TDDで始めましょう。 |
| 「テストが難しい = 設計が不明確」 | テストの声を聞きましょう。テストが難しい = 使用が難しい。 |
| 「TDDは遅くなる」 | TDDはデバッグよりも速い。プラグマティック = テストファースト。 |
| 「手動テストの方が速い」 | 手動ではエッジケースを証明できません。コード変更のたびに再テストが必要です。 |
| 「既存コードにはテストがない」 | あなたが改善する番です。既存コードにもテストを追加しましょう。 |

## 危険な兆候 - 停止してやり直す

*   テストより先にコードを書いた
*   実装後にテストを書いた
*   テストがすぐにパスした
*   なぜテストが失敗したか説明できない
*   テストが「後で」追加された
*   「今回だけは」と正当化する
*   「もう手動でテストした」
*   「後でテストしても同じ目的」
*   「精神の問題であって儀式ではない」
*   「参照として残す」または「既存コードを適応させる」
*   「すでにX時間費やした、削除するのは無駄だ」
*   「TDDはドグマだ、自分はプラグマティックだ」
*   「これは違うから…」

**これらすべては、コードを削除し、TDDでやり直すことを意味します。**

## バグ修正の例

**バグ:** 空のメールアドレスが受け入れられた

**RED**
```typescript
test('rejects empty email', async () => {
  const result = await submitForm({ email: '' });
  expect(result.error).toBe('Email required');
});
```

**Verify RED**
```bash
$ npm test
FAIL: expected 'Email required', got undefined
```

**GREEN**
```typescript
function submitForm(data: FormData) {
  if (!data.email?.trim()) {
    return { error: 'Email required' };
  }
  // ...
}
```

**Verify GREEN**
```bash
$ npm test
PASS
```

**REFACTOR**
必要に応じて、複数のフィールドのバリデーションを抽出する。

## 検証チェックリスト

作業完了とマークする前に:

*   [ ] すべての新しい関数/メソッドにテストがある
*   [ ] 実装する前に各テストが失敗するのを確認した
*   [ ] 各テストは期待された理由（機能の欠如、タイプミスではない）で失敗した
*   [ ] 各テストをパスさせるための最小限のコードを書いた
*   [ ] すべてのテストがパスする
*   [ ] 出力がきれいであること（エラー、警告がない）
*   [ ] テストは実際のコードを使用する（回避できない場合のみモックを使用）
*   [ ] エッジケースとエラーがカバーされている

すべてのチェックボックスにチェックを入れることができませんか？TDDをスキップしました。やり直してください。

## 行き詰まったとき

| 問題 | 解決策 |
|---------|----------|
| テスト方法が分からない | 望ましいAPIを書く。まずアサーションを書く。ユーザーに確認する。 |
| テストが複雑すぎる | 設計が複雑すぎる。インターフェースを簡素化する。 |
| 必須モック | コードの結合度が高すぎる。依存性注入を使用する。 |
| テストセットアップが大規模 | ヘルパーを抽出する。それでも複雑なら、設計を簡素化する。 |

## デバッグ統合

バグが見つかりましたか？それを再現する失敗するテストを書きましょう。TDDサイクルに従ってください。テストは修正を証明し、回帰を防ぎます。

テストなしでバグを修正しないでください。

## テストのアンチパターン

モックやテストユーティリティを追加する際は、よくある落とし穴を避けるために`@testing-anti-patterns.md`を読みましょう。
*   実際の挙動ではなく、モックの挙動をテストする
*   プロダクションクラスにテスト専用メソッドを追加する
*   依存関係を理解せずにモックする

## 最終ルール

```
プロダクションコード → テストが存在し、最初に失敗したこと
そうでなければ → TDDではない
```

ユーザーの許可なく例外を設けないでください。
