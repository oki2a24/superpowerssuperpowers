# Using Git Worktrees

## 概要

作業が隔離されたワークスペースで行われるようにします。プラットフォーム固有のネイティブなワークツリーツール（EnterWorktree等）がある場合はそれを優先し、利用可能なツールがない場合にのみ手動の git worktree にフォールバックします。

**コア原則:** 最初に既存の隔離環境を検出し、次にネイティブツールを使い、最後に git にフォールバックする。ハーネス（実行環境）に逆らわない。

**開始時の宣言:** 「隔離されたワークスペースをセットアップするために `using-git-worktrees` スキルを使用します。」

## ステップ 0: 既存の隔離環境の検出

**何かを作成する前に、すでに隔離されたワークスペースにいるかどうかを確認します：**

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
BRANCH=$(git branch --show-current)
```

**サブモジュール・ガード:** `GIT_DIR != GIT_COMMON` は git サブモジュールの内部でも真となります。「すでにワークツリーにいる」と結論付ける前に、サブモジュールではないことを確認してください：

```bash
# これがパスを返す場合、ワークツリーではなくサブモジュールの内部です — 通常のリポジトリとして扱います
git rev-parse --show-superproject-working-tree 2>/dev/null
```

**`GIT_DIR != GIT_COMMON` かつサブモジュールではない場合:** すでにリンクされたワークツリーにいます。ステップ 3（プロジェクトセットアップ）にスキップしてください。**新たにワークツリーを作成しないでください。**

ブランチの状態を報告します：
- ブランチ上にいる場合: 「すでに `<path>` の隔離されたワークスペース（ブランチ: `<name>`）にいます。」
- デタッチド HEAD の場合: 「すでに `<path>` の隔離されたワークスペース（デタッチド HEAD、外部管理）にいます。完了時にブランチ作成が必要です。」

**`GIT_DIR == GIT_COMMON`（またはサブモジュール内）の場合:** 通常のリポジトリ・チェックアウトにいます。

ユーザーがすでに指示書（GEMINI.md等）でワークツリーの好みを表明していますか？そうでない場合は、ワークツリーを作成する前に同意を得てください：

> 「隔離されたワークツリーをセットアップしましょうか？現在のブランチを変更から保護できます。」

すでに表明されている好みがある場合は、尋ねることなくそれに従います。ユーザーが同意しなかった場合は、現在の場所で作業を継続し、ステップ 3 に進みます。

## ステップ 1: 隔離されたワークスペースの作成

**2つのメカニズムがあります。この順序で試してください。**

### 1a. ネイティブ・ワークツリーツール (推奨)

ユーザーが隔離されたワークスペースを求めた場合（ステップ 0 の同意）、ワークツリーを作成する手段がすでに用意されていますか？ `EnterWorktree`, `WorktreeCreate`, `/worktree` コマンド、または `--worktree` フラグのような名前のツールがあるかもしれません。もしあるなら、それを使用してステップ 3 に進んでください。

ネイティブツールは、ディレクトリの配置、ブランチの作成、クリーンアップを自動的に処理します。ネイティブツールがあるのに `git worktree add` を使用すると、ハーネスが管理できない「幽霊のような状態」が生まれてしまいます。

利用可能なネイティブツールがない場合にのみ、ステップ 1b に進んでください。

### 1b. Git ワークツリーへのフォールバック

**ステップ 1a が適用されない場合にのみ使用してください** — 利用可能なネイティブツールがありません。git を使用して手動でワークツリーを作成します。

#### ディレクトリの選択

以下の優先順位に従ってください。明示的なユーザーの好みは、常にファイルシステムの状態よりも優先されます。

1. **指示書（GEMINI.md等）にワークツリーディレクトリの好みが記載されていないか確認します。** すでに指定されている場合は、尋ねることなくそれを使用します。

2. **既存のプロジェクトローカルなワークツリーディレクトリを確認します：**
   ```bash
   ls -d .worktrees 2>/dev/null     # 第一優先 (隠しディレクトリ)
   ls -d worktrees 2>/dev/null      # 第二優先
   ```
   見つかった場合、それを使用します。両方存在する場合は `.worktrees` が優先されます。

3. **既存のグローバルディレクトリを確認します：**
   ```bash
   project=$(basename "$(git rev-parse --show-toplevel)")
   ls -d ~/.gemini/worktrees/$project 2>/dev/null
   ```
   見つかった場合、それを使用します（以前のグローバルパスとの後方互換性）。

4. **他にガイドラインがない場合**、デフォルトとしてプロジェクトルートの `.worktrees/` を使用します。

#### 安全性検証 (プロジェクトローカルディレクトリのみ)

**ワークツリーを作成する前に、ディレクトリが無視されていることを必ず検証してください：**

```bash
git check-ignore -q .worktrees 2>/dev/null || git check-ignore -q worktrees 2>/dev/null
```

**無視されていない場合:** `.gitignore` に追加し、その変更をコミットしてから続行してください。

**重要性:** ワークツリーの内容を誤ってリポジトリにコミットするのを防ぎます。

グローバルディレクトリ (`~/.gemini/worktrees/`) の場合は、検証は不要です。

#### ワークツリーの作成

```bash
project=$(basename "$(git rev-parse --show-toplevel)")

# 選択された場所に基づいてパスを決定
# プロジェクトローカルの場合: path="$LOCATION/$BRANCH_NAME"
# グローバルの場合: path="~/.gemini/worktrees/$project/$BRANCH_NAME"

git worktree add "$path" -b "$BRANCH_NAME"
cd "$path"
```

**サンドボックス・フォールバック:** `git worktree add` が権限エラー（サンドボックスによる拒否）で失敗した場合は、ユーザーに「サンドボックスによりワークツリーの作成がブロックされたため、現在のディレクトリで作業を継続します」と伝え、その場所でセットアップとベースラインテストを実行してください。

## ステップ 3: プロジェクトセットアップ

適切なセットアップを自動検出し、実行します：

```bash
if [ -f package.json ]; then npm install; fi
if [ -f Cargo.toml ]; then cargo build; fi
if [ -f requirements.txt ]; then pip install -r requirements.txt; fi
if [ -f pyproject.toml ]; then poetry install; fi
if [ -f go.mod ]; then go mod download; fi
```

## ステップ 4: クリーンベースラインの検証

テストを実行し、ワークスペースが正常な状態で開始されたことを確認します：

```bash
# プロジェクトに適したコマンドを使用
npm test / cargo test / pytest / go test ./...
```

**テストが失敗した場合:** 失敗を報告し、続行するか調査するかを尋ねます。
**テストが成功した場合:** 準備完了を報告します。

### 報告

```text
ワークツリーの準備が完了しました: <full-path>
テスト成功 (<N> テスト通過, 0 失敗)
機能 <feature-name> の実装準備が整いました
```

## クイックリファレンス

| 状況 | アクション |
|-----------|--------|
| すでにリンクされたワークツリー内にいる | 作成をスキップ (ステップ 0) |
| サブモジュール内にいる | 通常のリポジトリとして扱う (ステップ 0 のガード) |
| ネイティブ・ワークツリーツールが利用可能 | それを使用する (ステップ 1a) |
| ネイティブツールがない | Git ワークツリーにフォールバック (ステップ 1b) |
| `.worktrees/` が存在 | それを使用 (無視されていることを検証) |
| `worktrees/` が存在 | それを使用 (無視されていることを検証) |
| 両方存在 | `.worktrees/` を使用 |
| どちらも存在しない | 指示書を確認し、デフォルトで `.worktrees/` を使用 |
| グローバルパスが存在 | それを使用 (後方互換性) |
| ディレクトリが無視されていない | .gitignore に追加 + コミット |
| 作成時に権限エラーが発生 | サンドボックス・フォールバック、その場で作業 |
| ベースライン中にテストが失敗 | 失敗を報告 + 尋ねる |
| package.json/Cargo.toml がない | 依存関係のインストールをスキップ |

## よくある間違い

### ハーネス（実行環境）に逆らう

- **問題:** プラットフォームがすでに隔離環境を提供しているのに `git worktree add` を使用する
- **修正:** ステップ 0 で既存の隔離を検出し、ステップ 1a でネイティブツールを優先する。

### 検出のスキップ

- **問題:** 既存のワークツリーの中に、さらにワークツリーを入れ子で作成してしまう
- **修正:** 何かを作成する前に必ずステップ 0 の検出を実行する。

### 無視検証のスキップ

- **問題:** ワークツリーの内容が追跡され、git status を汚染する
- **修正:** プロジェクトローカルな場所については必ず `git check-ignore` を実行する。

### ディレクトリの場所の仮定

- **問題:** 一貫性がなくなり、プロジェクトの慣習に反する
- **修正:** 優先順位に従う: 既存 > グローバル（レガシー） > 指示書 > デフォルト

### 失敗したテストで続行

- **問題:** 新しいバグと既存の問題を区別できない
- **修正:** 失敗を報告し、続行するための明示的な許可を得る

## レッドフラッグ (警告サイン)

### 決してしないこと (Never)
- ステップ 0 で既存の隔離環境が検出されたのにワークツリーを作成する
- ネイティブツール（`EnterWorktree`等）があるのに `git worktree add` を使用する。これは最大の間違いです。
- ステップ 1a を飛ばして、いきなりステップ 1b の git コマンドを実行する
- 無視されていることを検証せずにプロジェクトローカルなワークツリーを作成する
- ベースラインテストの検証をスキップする
- テストが失敗したまま、尋ねることなく続行する

### 常にすること (Always)
- 最初にステップ 0 の検出を実行する
- Git フォールバックよりもネイティブツールを優先する
- ディレクトリ選定の優先順位を遵守する: 既存 > グローバル（レガシー） > 指示書 > デフォルト
- プロジェクトローカルな場所について無視されていることを検証する
- プロジェクトのセットアップコマンドを自動検出して実行する
- クリーンなテストベースラインを検証する
