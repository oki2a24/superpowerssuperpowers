# 完了前の検証 (Verification Before Completion)

## 概要

検証なしに作業完了を主張することは、効率化ではなく不誠実な行為です。

**核心原則:** 常に、主張の前に証拠を。

**このルールの文字通りの意味に違反することは、その精神に違反することと同義です。**

## 鉄の掟

```
最新の検証証拠なしに、完了を主張してはならない
```

このメッセージの中で検証コマンドを実行し、その結果を確認していないのであれば、それがパスしたと主張することはできません。

## ゲート機能 (Gate Function)

状態を主張したり、満足感（「できました！」「完璧です！」など）を表現したりする**前に**:

1. **特定:** その主張を証明するコマンドは何か？（例: `npm test`, `pytest`, `git diff`）
2. **実行:** コマンドを完全に（最新の状態で、省略せずに）実行する。
3. **読み取り:** 出力をすべて読み、終了コード、エラー数、失敗数を確認する。
4. **検証:** 出力は主張を裏付けているか？
   - **NO の場合:** 証拠と共に実際の状態を報告する。
   - **YES の場合:** 証拠（出力の抜粋など）を添えて、成功を主張する。
5. **その後にのみ:** 完了を宣言する。

ステップを一つでも飛ばすことは、検証ではなく「嘘」をついていることになります。

## よくある失敗例

| 主張 | 必要なもの | 不十分な例 |
|-------|----------|----------------|
| テストをパスした | テストコマンドの出力: 失敗 0 | 前回の実行結果、「パスするはず」という推測 |
| リンターがクリーン | リンターの出力: エラー 0 | 部分的なチェック、外挿 |
| ビルドに成功 | ビルドコマンド: exit 0 | リンターがパスした、ログが良さそうに見える |
| バグを修正した | 元の症状の再現テスト: パス | コードを変更した、直ったと想定した |
| 回帰テストが機能 | レッド・グリーンサイクルの検証 | 一回だけテストがパスした |
| エージェントが完了した | VCS (git) の差分が変更を示している | エージェントが「成功」と報告した |
| 要件を満たした | 行ごとのチェックリスト | テストがパスしただけ |

## レッドフラグ - 即座に停止

- 「はず」「たぶん」「〜と思われる」といった言葉の使用。
- 検証前に満足感を表現すること（「素晴らしい！」「完璧！」「終わりました！」など）。
- 検証せずにコミット、プッシュ、プルリクエスト（PR）を作成しようとすること。
- エージェントの成功報告を鵜呑みにすること。
- 部分的な検証に頼ること。
- 「今回だけは特別」と考えること。
- 疲れていて、早く作業を終わらせたいと思うこと。
- **検証を実行せずに、成功を暗示するあらゆる表現。**

## 正当化の防止

| 言い訳 | 現実 |
|--------|---------|
| 「今は動くはずです」 | 検証を実行してください |
| 「自信があります」 | 自信は証拠ではありません |
| 「今回だけです」 | 例外はありません |
| 「リンターは通りました」 | リンターはコンパイラではありません |
| 「エージェントが成功と言っています」 | 独立して検証してください |
| 「疲れています」 | 疲労は言い訳になりません |
| 「部分的なチェックで十分です」 | 部分的では何も証明できません |
| 「言葉が違うからルールは適用されない」 | 文言ではなく精神を優先してください |

## 主要なパターン

**テスト:**
```
✅ [テストコマンド実行] [結果: 34/34 パス] 「すべてのテストをパスしました」
❌ 「今はパスするはずです」 / 「正しく見えます」
```

**回帰テスト (TDD レッド・グリーン):**
```
✅ 作成 → 実行 (パス) → 修正を戻す → 実行 (必ず失敗することを確認) → 修正を戻す → 実行 (パス)
❌ 「回帰テストを書きました」（レッド・グリーン検証なし）
```

**ビルド:**
```
✅ [ビルド実行] [結果: exit 0] 「ビルドがパスしました」
❌ 「リンターが通りました」（リンターはコンパイルの成否をチェックしません）
```

**要件:**
```
✅ 計画を再読 → チェックリスト作成 → 各項目を検証 → ギャップまたは完了を報告
❌ 「テストが通ったので、フェーズ完了です」
```

## なぜこれが重要か

不誠実な報告が招く結果：
- ユーザーの信頼を失う。
- 未定義の関数が含まれたまま出荷され、クラッシュする。
- 欠落した要件が出荷され、機能が不完全になる。
- 虚偽の完了報告により、手戻りが発生し時間が無駄になる。

## 適用時期

**以下の前には必ず適用すること:**
- 成功や完了を主張するあらゆるバリエーション。
- 満足感の表現。
- 作業の状態に関するポジティブな発言。
- コミット、PR 作成、タスク完了。
- 次のタスクへの移行。
- 他のエージェントへの委任。

## 結論

**検証にショートカットはありません。**

コマンドを実行し、出力を読み、それから結果を主張してください。

これは交渉の余地のない、必須のルールです。
