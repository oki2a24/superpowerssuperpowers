# 計画の作成 (Writing Plans)

## 概要

エンジニアがコードベースの知識を全く持たず、実装の詳細についても指示が必要であることを前提として、包括的な実装計画を作成します。各タスクでどのファイルを変更するか、コード例、テスト方法、確認すべきドキュメントなど、必要なすべての情報を記述します。計画は一口サイズのタスクに分割して提供されます。DRY（Don't Repeat Yourself）、YAGNI（You Aren't Gonna Need It）、TDD（Test-Driven Development）の原則に従い、頻繁なコミットを推奨します。

開発者は熟練しているものの、使用するツールセットや問題領域についてはほとんど知らないと仮定します。また、優れたテスト設計についてもあまり知らないと仮定します。

**開始時の宣言:** 「実装計画を作成するために writing-plans スキルを使用します。」

**コンテキスト:** このスキルは（brainstorming スキルによって作成された）専用のワークツリー内で実行されるべきです。

**計画の保存先:** `docs/plans/YYYY-MM-DD-<feature-name>.md`
- (計画の場所に関するユーザー設定がある場合は、そちらを優先します)

## 範囲チェック (Scope Check)

仕様書が複数の独立したサブシステムにわたっている場合は、brainstorming の段階でサブプロジェクトの仕様に分割されているべきです。そうなっていない場合は、サブシステムごとに個別の計画を立てることを提案してください。各計画は、それ自体で動作し、テスト可能なソフトウェアを生成する必要があります。

## ファイル構成 (File Structure)

タスクを定義する前に、どのファイルを作成または変更し、それぞれが何を担当するかを計画します。ここで設計上の決定が確定されます。

- 境界が明確で、定義されたインターフェースを持つ設計単位にします。各ファイルには1つの明確な責任を持たせます。
- 一度にコンテキストに保持できるコードについては、より正確に推論できます。ファイルが焦点を絞っているとき、修正はより信頼性が高くなります。機能が多すぎる大規模なファイルよりも、小さく焦点を絞ったファイルを優先してください。
- 共に変更されるファイルは共に配置されるべきです。技術的なレイヤーではなく、責任ごとに分割します。
- 既存のコードベースでは、確立されたパターンに従います。コードベースで大規模なファイルが使用されている場合は、一方的に再構成しないでください。ただし、変更しようとしているファイルが手に負えないほど大きくなっている場合は、計画の中で分割を検討するのが妥当です。

このファイル構成が、タスク分解の直接的な指針となります。**各タスクは、独立して意味を成し、それ自体が検証可能な「自己完結型」の変更を生成しなければなりません。**

## 計画文書ヘッダー (Plan Document Header)

すべての計画文書は、以下のヘッダーから開始する必要があります。

```markdown
# [機能名] 実装計画

> **AIエージェントへの指示:** REQUIRED SUB-SKILL: この計画をタスクごとに実装するには、移植された `subagent-driven-development` スキル（推奨）または `executing-plans` スキルを `activate_skill` で起動して使用してください。ステップには追跡用のチェックボックス (`- [ ]`) を使用します。

**目標:** [この計画で構築する内容を説明する一文]

**アーキテクチャ:** [アプローチに関する2-3文]

**技術スタック:** [主要な技術/ライブラリ]

---
```

## 一口サイズのタスク粒度 (Bite-Sized Task Granularity)

各ステップは1つのアクション（2-5分）にします。

- "失敗するテストを作成" - ステップ
- "テストが失敗することを確認するために実行" - ステップ
- "最小限の実装を作成" - ステップ
- "テストがパスすることを確認するために実行" - ステップ
- "コミット" - ステップ

## タスク構造 (Task Structure)

````markdown
### タスク N: [コンポーネント名]

**ファイル:**
- 作成: `exact/path/to/file.py`
- 変更: `exact/path/to/existing.py:123-145`
- テスト: `tests/exact/path/to/test.py`

- [ ] **ステップ 1: 失敗するテストを作成**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **ステップ 2: テストが失敗することを確認するために実行**

実行: `pytest tests/path/test.py::test_name -v`
期待値: "function not defined"でFAIL

- [ ] **ステップ 3: 最小限の実装を作成**

```python
def function(input):
    return expected
```

- [ ] **ステップ 4: テストがパスすることを確認するために実行**

実行: `pytest tests/path/test.py::test_name -v`
期待値: PASS

- [ ] **ステップ 5: コミット**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## プレースホルダーの禁止 (No Placeholders)

各ステップには、開発者が必要とする実際の内容を含める必要があります。以下のようなパターンは**計画の失敗**であり、決して記述してはいけません。

- "TBD"、"TODO"、"後で実装"、"詳細を記入"
- "適切なエラー処理を追加" / "検証を追加" / "エッジケースを処理"
- "上記のテストを作成"（実際のテストコードがない場合）
- "タスクNと同様"（コードを繰り返してください。開発者はタスクを順不同で読んでいる可能性があります）
- コードの書き方を示さずに何をするかだけを説明するステップ（コードステップにはコードブロックが必須）
- どのタスクでも定義されていない型、関数、またはメソッドへの参照

## 覚えておくべきこと (Remember)

- 常に正確なファイルパスを使用すること。
- 各ステップに完全なコードを含めること。コードを変更する場合は、そのコードを示すこと。
- 期待される出力を含む正確なコマンドを記述すること。
- DRY、YAGNI、TDD、頻繁なコミットの原則に従うこと。

## セルフレビュー (Self-Review)

計画全体を書き終えた後、新鮮な目で仕様書を見直し、以下のチェックリストを**自分自身で**実行してください。これはサブエージェントのディスパッチ前に行う「自己防衛」です。

1. **仕様の網羅性**: 仕様書の各要件に対し、それを実装するタスクを特定できるか？
2. **プレースホルダーのスキャン**: 「TBD」、「TODO」、「詳細を記入」などの曖昧な表現はないか？
3. **型の整合性**: 後のタスクで使用する型、メソッド名、プロパティ名が、以前のタスクで定義したものと一致しているか？（例：タスク3で `clearLayers()`、タスク7で `clearFullLayers()` と呼ぶのはバグ）。
4. **コマンドの正確性**: 記述したコマンドがそのディレクトリで実際に動作するか？

問題が見つかった場合は、その場で修正してください。再レビューは不要です。

### サブエージェントによる検証 (Subagent Review)

計画が大規模、または複雑な場合は、自分自身のセルフレビューに加えて、サブエージェントによる客観的な検証を行うことを強く推奨します。

- **手順:** `plan-document-reviewer-prompt.md` のテンプレートを使用し、仕様書と作成した計画書を渡してレビュー用サブエージェントをディスパッチします。
- **目的:** 計画の完全性、仕様との整合性、およびタスク分解の妥当性を第三者の視点で確認します。

## 実行の引き渡し (Execution Handoff)

計画を保存した後、実行オプションを提示します。

**「計画が完成し、`docs/plans/<filename>.md` に保存されました。2つの実行オプションがあります:**

**1. サブエージェント駆動 (推奨)** - タスクごとに新しいサブエージェントをディスパッチし、タスク間でレビューを行い、迅速なイテレーションを実現します。

**2. インライン実行** - このセッション内で `executing-plans` を使用して、チェックポイントを設けながらタスクを一括実行します。

**どちらのアプローチで実行しますか？」**

**サブエージェント駆動が選択された場合:**
- **必須サブスキル:** `subagent-driven-development` を使用します。
- タスクごとに新しいサブエージェントをディスパッチし、2段階のレビューを行います。

**インライン実行が選択された場合:**
- **必須サブスキル:** `executing-plans` を使用します。
- レビュー用のチェックポイントを設けながら、タスクを一括実行します。
