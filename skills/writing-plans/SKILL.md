# 計画の作成

## 概要

実装計画を作成します。この計画は、コードベースの知識がゼロで、実装の詳細について指示が必要な開発者を想定して作成されます。各タスクでどのファイルを変更するか、コード例、テスト方法、確認すべきドキュメントなど、必要なすべての情報を記述します。計画は一口サイズのタスクに分割して提供されます。DRY（Don't Repeat Yourself）、YAGNI（You Aren't Gonna Need It）、TDD（Test-Driven Development）の原則に従い、頻繁なコミットを推奨します。

開発者は熟練しているものの、使用するツールセットや問題領域についてはほとんど知らないと仮定します。また、優れたテスト設計についてもあまり知らないと仮定します。

## プロセス

### 1. 計画のコンテキスト設定

*   **AIエージェントへの指示**: ユーザーに「計画を作成する機能の名称」と「計画の基となる仕様または要件の記述」を尋ねてください。取得後、「機能名: [取得した機能名]、要件: [取得した要件]で計画を作成します。次のステップに進むかユーザーに確認してください。」と報告してください。
*   **AIエージェントへの指示**: このスキルは専用のワークツリー内で実行されることを想定しています。もし必要であれば、`using-git-worktrees`スキルを使ってワークツリーを作成することをユーザーに提案してください。
*   **計画の保存先**: 計画は`docs/plans/YYYY-MM-DD-<feature-name>.md`に保存されます。

### 2. 計画文書ヘッダーの作成

すべての計画文書は、以下のヘッダーから開始する必要があります。

```markdown
# [機能名] 実装計画

> **AIエージェントへの指示**: この計画をタスクごとに実装するには、移植された`executing-plans`スキルを使用してください。

**目標:** [この計画で構築する内容を説明する一文]

**アーキテクチャ:** [アプローチに関する2-3文]

**技術スタック:** [主要な技術/ライブラリ]

---
```

### 3. 一口サイズのタスク粒度

各ステップは1つのアクション（2-5分）です。

*   "失敗するテストを作成" - ステップ
*   "テストが失敗することを確認するために実行" - ステップ
*   "最小限の実装を作成" - ステップ
*   "テストがパスすることを確認するために実行" - ステップ
*   "コミット" - ステップ

### 4. タスク構造

```markdown
### タスク N: [コンポーネント名]

**ファイル:**
- 作成: `exact/path/to/file.py`
- 変更: `exact/path/to/existing.py:123-145`
- テスト: `tests/exact/path/to/test.py`

**ステップ 1: 失敗するテストを作成**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

**ステップ 2: テストが失敗することを確認するために実行**

実行: `pytest tests/path/test.py::test_name -v`
期待値: "function not defined"でFAIL

**ステップ 3: 最小限の実装を作成**

```python
def function(input):
    return expected
```

**ステップ 4: テストがパスすることを確認するために実行**

実行: `pytest tests/path/test.py::test_name -v`
期待値: PASS

**ステップ 5: コミット**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
```

### 5. 注意事項

*   常に正確なファイルパスを使用すること。
*   計画には完全なコードを含め、「検証を追加」のような曖昧な表現は避けること。
*   期待される出力を含む正確なコマンドを記述すること。
*   関連するスキル（`activate_skill`）を参照すること。
*   DRY, YAGNI, TDD, 頻繁なコミットの原則に従うこと。

### 6. 実行の引き渡し

計画を保存した後、実行オプションを提示します。

**「計画が完成し、`docs/plans/<filename>.md`に保存されました。実行オプションは二つあります:**

**1. サブエージェント駆動 (このセッション内)** - タスクごとに新しいサブエージェントをディスパッチし、タスク間でレビューを行い、迅速なイテレーションを実現します。

**2. 並行セッション (別のセッション)** - `executing-plans`スキルを使用して新しいセッションを開き、チェックポイントを持つバッチ実行を行います。

**どちらのアプローチで実行しますか？」**

*   **サブエージェント駆動が選択された場合:**
    *   **AIエージェントへの指示**: 移植された`subagent-driven-development`スキルを`activate_skill`で起動してください。
    *   このセッション内で作業を続行し、タスクごとに新しいサブエージェントをディスパッチし、コードレビューを行います。
*   **並行セッションが選択された場合:**
    *   ユーザーに、ワークツリー内で新しいセッションを開くように指示してください。
    *   **AIエージェントへの指示**: 新しいセッション内で移植された`executing-plans`スキルを`activate_skill`で起動してください。

### AIエージェントへの指示 (Gemini固有)

1.  **物理的 I/O 定義の義務化 (Mandatory for Code Changes)**:
    *   プログラムの修正や新規作成を伴うタスクには、以下の項目を「物理的・具体的」に定義することを**義務**付けます。
    *   **入力**: 引数名と型、期待される環境変数、読み込むファイルパス。
    *   **出力**: 戻り値、標準出力/エラーの具体的な形式、生成・更新されるファイルのパスとその完全なテンプレート（または内容）。
    *   設計書への参照で済ませず、計画書単体で実装が完結する状態（Self-contained）にしてください。

2.  **テンプレートの完全転写**:
    *   生成すべきファイルの完全なテンプレートや標準出力のレイアウトを、計画書内に直接「物理的にコピー」して掲載してください。

3.  **役割分担の明示 (Hybrid Pattern)**:
    *   スクリプト（構造生成）と AI（文脈記入）の役割分担がある場合は、どのステップでどちらが責任を持つかを明確に記述してください。

4.  **セッション分割の判定基準 (Mandatory Partitioning Check)**:
    *   計画策定の最終段階で、以下のいずれかに該当する場合は**必ず** `session-coordination` スキルによる分割をユーザーに提案してください。
        *   **規模**: タスク（Task）数が 5つ以上、または合計ステップ数が 20以上。
        *   **健康状態**: 現在のセッションが既に 15ターン以上経過している。
        *   **独立性**: 異なるモジュールや、依存関係のない並行可能な作業が含まれる。

5.  **GPAC v3 へのハンドオフ手順 (Execution Handoff Refinement)**:
    *   オリジナルの「6. 実行の引き渡し」において「並行セッション (別のセッション)」が選択された場合、または上記 4 の基準を満たす場合、以下の手順を**厳格に実行**してください。
        1.  `activate_skill(name="session-coordination")` を呼び出す。
        2.  計画書の各タスクを「独立した指示書（Markdown）」として一時ファイルに書き出す。
        3.  `gemini-sub spawn` コマンドを生成し、ユーザーに提示（または実行許可を求める）。
        4.  サブセッション側では、配布された指示書に基づき `activate_skill(name="executing-plans")` を起動して作業を開始するよう明記する。

### AIエージェントへの指示 (Gemini固有)
- **[イメージ] 図解の規律 (Visual Discipline)**: 複雑なタスクの依存関係やデータ処理フローを記述する際は、必ず以下の AA テンプレートを参考に構造（状態遷移、データフロー、ディレクトリ構成等）を可視化せよ。

#### 1. 状態遷移図 (State Transition)
複雑なオブジェクトのライフサイクル（例：タスクのステータス遷移）に使用。
```text
( Initial ) --[ Start ]--> / Running / --[ Success ]--> [ Completed ]
                             ^    |
                             |    v
                             | [ Failure ] --> [ Error ]
                             |                   |
                             +----[ Retry ]------+
```

#### 2. 直列・分岐フロー (Data Flow / Process)
処理の順序や条件分岐（例：エラーハンドリング）に使用。
```text
          +--( If: A )--> [ Process A ] --+
[ Input ] |                               |--> [ Output ]
          +--( If: B )--> [ Process B ] --+
```

#### 3. 構造図 (Structure / Hierarchy)
ファイルディレクトリ構成や、データの親子関係に使用。
```text
Root/
├── parent/
│   ├── child_A (Status: [/])
│   └── child_B (Status: [ ])
└── common/
```
