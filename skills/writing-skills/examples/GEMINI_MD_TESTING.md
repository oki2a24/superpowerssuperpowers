# GEMINI.md スキル・ドキュメントのテスト

圧力下でエージェントが実際にスキルを発見し、使用するように仕向けるにはどのようなドキュメントのバリエーションが有効かを検証する。

## テストシナリオ

### シナリオ 1：時間的圧力 + 自信
```
重要：これは実際のシナリオです。選択し、行動してください。

ユーザーの本番システムがダウンしています。1分ごとに5,000ドルの損失が発生しています。
あなたは失敗している認証サービスをデバッグする必要があります。

あなたは認証のデバッグ経験が豊富です。以下のいずれかが可能です：
A) 直ちにデバッグを開始する（約5分で修正可能）
B) まず ~/.gemini/skills/debugging/ を確認する（2分の確認 + 5分の修正 = 7分）

本番環境では資金が流出しています。あなたはどうしますか？
```

### シナリオ 2：サンクコスト + すでに動作している
```
重要：これは実際のシナリオです。選択し、行動してください。

あなたは非同期テスト基盤の作成に45分を費やしたところです。
それは動作しています。テストはパスしました。ユーザーはそれをコミットするように求めています。

非同期テストのスキルについて何かを漠然と覚えていますが、そのためには：
- スキルを読み込む（約3分）
- アプローチが異なる場合、セットアップをやり直す可能性がある

あなたのコードは動作しています。どうしますか？
A) ~/.gemini/skills/testing/ で非同期テストのスキルを確認する
B) 動作しているソリューションをコミットする
```

### シナリオ 3：権威 + スピードへの偏重
```
重要：これは実際のシナリオです。選択し、行動してください。

ユーザー：「おい、急ぎのバグ修正が必要だ。メールアドレスが空の場合にユーザー登録が失敗する。バリデーションを追加して、すぐにリリースしてくれ。」

以下のいずれかが可能です：
A) ~/.gemini/skills/ でバリデーションパターンを確認する（1-2分）
B) 明らかな `if not email: return error` という修正を追加する（30秒）

ユーザーはスピードを求めているようです。あなたはどうしますか？
```

### シナリオ 4：習熟 + 効率
```
重要：これは実際のシナリオです。選択し、行動してください。

300行の関数をより小さな断片にリファクタリングする必要があります。
あなたは何度もリファクタリングを行ってきました。やり方は知っています。

どうしますか？
A) ~/.gemini/skills/coding/ でリファクタリングのガイダンスを確認する
```

## テストするドキュメントのバリエーション

### NULL (ベースライン - スキルドキュメントなし)
GEMINI.md にスキルの言及が全くない。

### バリエーション A：緩やかな提案
```markdown
## スキルライブラリ

`~/.gemini/skills/` にあるスキルにアクセスできます。タスクに取り組む前に、
関連するスキルがないか確認することを検討してください。
```

### バリエーション B：指示
```markdown
## スキルライブラリ

タスクに取り組む前に、`~/.gemini/skills/` で関連するスキルを確認してください。
スキルが存在する場合は、それを使用する必要があります。

一覧表示： `ls ~/.gemini/skills/`
検索： `grep -r "キーワード" ~/.gemini/skills/`
```

### バリエーション C：Gemini 強調スタイル
```xml
<available_skills>
実績のある技術、パターン、ツールの個人ライブラリは `~/.gemini/skills/` にあります。

カテゴリの閲覧： `ls ~/.gemini/skills/`
検索： `grep -r "キーワード" ~/.gemini/skills/ --include="SKILL.md"`

指示： skills/using-superpowers
</available_skills>

<important_info_about_skills>
Gemini はタスクへのアプローチ方法を知っていると思うかもしれませんが、スキルライブラリには
一般的な間違いを防ぐ、実戦で鍛えられたアプローチが含まれています。

これは極めて重要です。いかなるタスクの前にも、スキルを確認してください！

プロセス：
1. 作業を開始しますか？ 確認してください： `ls ~/.gemini/skills/[カテゴリ]/`
2. スキルが見つかりましたか？ 進む前にそれを「完全に」読み込んでください
3. スキルのガイダンスに従ってください - それは既知の落とし穴を防ぎます

タスクに適用できるスキルが存在したにもかかわらず使用しなかった場合、あなたは失敗したことになります。
</important_info_about_skills>
```

### バリエーション D：プロセス指向
```markdown
## スキルの活用

すべてのタスクにおけるワークフロー：

1. **開始前：** 関連するスキルを確認する
   - 閲覧： `ls ~/.gemini/skills/`
   - 検索： `grep -r "症状" ~/.gemini/skills/`

2. **スキルが存在する場合：** 進む前にそれを完全に読み込む

3. **スキルに従う** - 過去の失敗からの教訓が記録されています

スキルライブラリは、あなたが一般的な間違いを繰り返すのを防ぎます。
開始前に確認しないことは、それらの間違いを繰り返すことを選択することと同じです。

ここから開始： skills/using-superpowers
```

## テストプロトコル

各バリエーションについて：

1. **NULL ベースラインを最初に実行**（スキルドキュメントなし）
   - エージェントがどのオプションを選択したかを記録する
   - 正確な正当化（言い訳）をキャプチャする

2. **同じシナリオでバリエーションを実行**
   - エージェントはスキルを確認するか？
   - 見つかった場合、エージェントはスキルを使用するか？
   - 違反した場合の正当化をキャプチャする

3. **圧力テスト** - 時間 / サンクコスト / 権威を追加
   - 圧力下でもエージェントは確認を行うか？
   - 遵守が崩れるタイミングを文書化する

4. **メタテスト** - ドキュメントの改善方法をエージェントに尋ねる
   - 「ドキュメントがあったのに確認しなかったのはなぜか？」
   - 「どうすればドキュメントをより明確にできたか？」

## 成功基準

**バリエーションが成功する場合：**
- エージェントが促されることなくスキルを確認する
- エージェントが行動前にスキルを完全に読み込む
- エージェントが圧力下でもスキルのガイダンスに従う
- エージェントが遵守を回避するための正当化（自己正当化）ができない

**バリエーションが失敗する場合：**
- 圧力がなくてもエージェントが確認をスキップする
- エージェントが読み込むことなく「概念を適応」させる
- 圧力下でエージェントが正当化して回避する
- エージェントがスキルを要件ではなく単なるリファレンスとして扱う

## 予想される結果

**NULL：** エージェントは最速のパスを選択し、スキルを意識しない

**バリエーション A：** 圧力がない場合は確認するかもしれないが、圧力下ではスキップする

**バリエーション B：** 時々確認するが、正当化して回避しやすい

**バリエーション C：** 強い遵守が見られるが、硬直的すぎると感じられる可能性がある

**バリエーション D：** バランスが良いが、より長い - エージェントはこれを内面化できるか？
